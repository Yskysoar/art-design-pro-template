# 后端开发设计方案

> 本文档基于当前 `Docs/` 文档与 `Code/art-design-pro` 前端代码整理，用于指导后续 Java + Spring Boot + MySQL + MyBatis-Plus 后端开发。当前已在 `Code/backend-template-java` 初始化后端骨架。

最后更新：2026-06-01

## 一、设计目标

后端项目需要作为当前 Vue 管理后台模板的真实业务服务，替换开发期 Mock，并满足后续上线要求。后端工程存放在 `Code/backend-template-java`，当前先按单体分模块方式建设，后期如果模块边界和业务规模需要，再升级为 Maven 多模块单体。

核心目标：

- 与前端现有接口格式、权限模型、菜单结构和页面模块自洽对接。
- 后端模块保持独立，每个模块可单独进行单元测试、接口测试和权限测试。
- 上线前具备基础安全防护：认证、授权、XSS 防护、上传校验、操作审计、异常处理和生产配置隔离。
- 使用 Java、Spring Boot、MySQL、MyBatis-Plus 作为主要技术栈。
- 使用 Maven 构建，Java 版本为 17，Spring Boot 使用 3.x。
- 遵循 Spring 三层架构：Controller 只处理请求、响应和参数规范，真实业务逻辑放在 Service 层。
- 使用 MyBatis-Plus 时优先使用内置 CRUD、Wrapper、分页、逻辑删除和自动填充能力；只有复杂查询、复杂统计或复杂函数场景才自定义 SQL。
- 接口路径统一使用 `/api/**`。
- 认证方案使用 Bearer JWT，accessToken 有效期为 30 分钟，初期不实现 refreshToken。
- 权限模型使用 RBAC + 组织树数据权限，并支持前台 + 后台双端权限模型。

### 2.3 权限模式

当前前端支持 `frontend` 和 `backend` 两种权限模式。正式后端开发建议使用：

```env
VITE_ACCESS_MODE=backend
```

当前项目默认使用 `backend` 权限模式，`.env.development` 中应保持 `VITE_ACCESS_MODE=backend`。后端需要返回当前登录用户可访问的菜单树，并在接口层继续做服务端权限校验。前端按钮权限来自用户信息 `buttons` 和菜单 `meta.authList`，不能作为唯一安全边界。当前系统管理主要接口已接入 `PermissionService` 权限码校验，超级管理员 `R_SUPER` 需要同时存在于 JWT 角色声明和数据库启用角色数据中才可跳过权限码校验，普通角色必须拥有对应权限码。

前端动态路由初始化需要区分权限模式：`backend` 模式下，`GET /api/user/info` 与 `GET /api/v3/system/menus` 可以并发请求，减少首次页面跳转等待；`frontend` 模式下，本地菜单过滤依赖 `user.info.roles`，必须先获取用户信息再生成菜单。路由初始化期间应复用同一个初始化任务，并只恢复最新一次导航，避免快速切换页面时重复请求、旧导航反抢新导航或误判 token 失效。

401 处理仍代表未登录或 token 失效，前端必须退出登录；但同一轮并发请求产生的多个 401 只应触发一次退出、一次动态路由重置和一次登录页跳转。403 代表权限不足，只提示无权限或进入 403 流程，不清理 token。

## 四、设计选型

### 4.1 后端工程形态

| 方案 | 说明 | 优点 | 代价 | 适用场景 |
| :--- | :--- | :--- | :--- | :--- |
| A. 单体分模块 | 一个 Spring Boot 应用，按包拆分模块 | 开发快、部署简单、适合当前阶段 | 模块边界依赖团队自律 | 推荐起步方案 |
| B. Maven 多模块单体 | parent + common + system + article + file 等模块 | 模块边界更清晰，利于测试和复用 | 初期配置更多 | 业务会持续扩展时推荐 |
| C. 微服务 | 认证、系统、文章、文件独立服务 | 扩展性强 | 运维、网关、链路追踪复杂 | 当前不建议起步使用 |

已确认：当前采用 A，单体分模块。后期如果系统模块增多、复用边界稳定，再升级为 B，Maven 多模块单体。当前不采用微服务。

### 4.2 认证方案

| 方案 | 前端改动 | 安全性 | 后端复杂度 | 说明 |
| :--- | :--- | :--- | :--- | :--- |
| A. Bearer JWT | 小。把请求头统一为 `Authorization: Bearer <token>` | 中 | 中 | 前后端分离常见，需处理 token 过期、刷新、黑名单 |
| B. HttpOnly Cookie Session | 中。前端开启 `withCredentials`，移除 JS 可读 token | 高 | 中 | 更抗 XSS 窃取 token，需 CSRF 防护和跨域配置 |
| C. 短 token + 刷新 token Cookie | 中高 | 高 | 高 | accessToken 短期有效，refreshToken 放 HttpOnly Cookie |

已确认：当前采用 A，Bearer JWT。后端返回 accessToken，前端请求头统一使用：

```text
Authorization: Bearer <accessToken>
```

accessToken 有效期为 30 分钟，初期不实现 refreshToken。后续如果上线安全要求提高，可以升级为短 accessToken + HttpOnly Cookie refreshToken。

### 4.3 权限模型

| 方案 | 说明 | 优点 | 代价 |
| :--- | :--- | :--- | :--- |
| A. RBAC | 用户-角色-菜单-按钮权限 | 简洁，和当前前端最匹配 | 复杂数据权限需要扩展 |
| B. RBAC + 数据权限 | 在 RBAC 基础上增加部门、本人、本部门及子部门等数据范围 | 适合管理后台 | 表结构和查询条件更复杂 |
| C. ABAC | 按属性和策略动态判断 | 灵活 | 设计、测试、排错成本高 |

已确认：当前使用 B，RBAC + 组织树数据权限。

权限模型需要支持通用可配置项目，覆盖单后台系统、前台 + 后台系统、博客系统、社团/部门/组织管理系统、商家系统等场景。

角色不硬编码业务身份，使用角色配置承载层级、访问端和管理限制：

```text
role_code       角色编码，例如 R_SUPER、R_MODERATOR、R_USER、R_GUEST
role_level      角色层级数值，数值越高管理级别越高
role_sort       展示排序
role_type       SYSTEM / BUSINESS / GUEST
access_scope    ADMIN / PORTAL / BOTH
manageable      是否允许被非超级管理员管理
```

角色示例：

```text
超级管理员  role_level=100  access_scope=ADMIN 或 BOTH
部长        role_level=60   access_scope=BOTH
商家        role_level=50   access_scope=BOTH
商家管理员  role_level=40   access_scope=BOTH
普通用户    role_level=10   access_scope=PORTAL
游客        role_level=0    access_scope=PORTAL
```

角色层级用于表达不同业务系统中的管理关系和权限边界，例如“部长不能创建超级管理员”“商家管理员不能管理其他商家”“场馆拥有者可以设置多个场馆管理员”。接口鉴权仍以权限码为准，例如：

```text
system:user:add
system:role:edit
org:member:manage
merchant:shop:edit
article:publish:add
```

模板提供通用角色层级字段和基础校验能力，但不固定某一种业务层级模型。具体层级设计由超级管理员按项目配置和维护，例如场馆系统可配置场馆拥有者、场馆管理员、预定用户等关系；社团系统可配置部长、社团管理员、成员等关系。角色层级、角色权限、用户组织关联模式等基础规则只能由超级管理员维护。

### 4.4 菜单返回方式

| 方案 | 说明 | 优点 | 代价 |
| :--- | :--- | :--- | :--- |
| A. 后端直接返回前端路由结构 | 返回 `path/name/component/meta/children` | 前端改动最小 | 后端需要理解前端路由字段 |
| B. 后端返回标准菜单模型，前端转换路由 | 后端只返回业务菜单字段 | 后端更纯粹 | 前端需要新增转换层 |

已确认：当前采用 A，后端直接返回前端路由结构，保证快速替换 Mock，并兼容当前 `VITE_ACCESS_MODE=backend`。

### 4.5 文件存储方案

| 方案 | 说明 | 优点 | 风险/代价 |
| :--- | :--- | :--- | :--- |
| A. 本地磁盘 | 文件存到服务器本地目录 | 简单，适合开发 | 多实例、备份、权限控制较麻烦 |
| B. MinIO | 自建对象存储 | 接近生产对象存储，成本可控 | 需要额外服务 |
| C. 云对象存储 | 使用云厂商 OSS/COS/S3 | 稳定，适合线上 | 需要账号、费用、回调安全设计 |

已确认：当前只采用 A，本地磁盘存储，暂不考虑 MinIO、云对象存储或其他部署形态。文件存储模块保留清晰 Service 边界，便于未来项目需要时再扩展，但当前模板开发不提前实现多存储适配。

### 4.6 富文本安全方案

| 方案 | 说明 | 优点 | 代价 |
| :--- | :--- | :--- | :--- |
| A. 保存前净化 | 后端入库前过滤危险 HTML | 读取快，数据更干净 | 规则变更后需重洗历史数据 |
| B. 输出前净化 | 返回前过滤危险 HTML | 策略可调整 | 每次读取有额外成本 |
| C. 保存前 + 输出前双净化 | 双保险 | 安全性最好 | 实现和测试更多 |

已确认：采用 C，保存前 + 输出前双净化。前端仍需要 DOMPurify 等净化措施，后端不能把前端当作唯一防线。

### 4.7 前台 + 后台双端模型

项目需要同时支持两种运行形态：

| 形态 | 说明 |
| :--- | :--- |
| 单后台 | 只有后台管理端，例如内部管理系统。 |
| 前台 + 后台 | 有公开前台和管理后台，例如博客、社团展示、商家系统。 |

建议用访问端范围区分菜单、路由、接口和角色：

```text
ADMIN   后台管理端
PORTAL  前台业务端
BOTH    前后台都可访问
```

游客是前台预览业务角色，只允许匿名访问前台公开内容，不能进入后台，不能执行评论、点赞、收藏、发布、上传等写操作。系统不提供匿名发布能力。普通登录用户可以根据业务权限执行前台交互；版主、商家管理员等角色可以同时访问前台和对应后台管理功能。

前台和后台共用同一套认证后端。前端登录成功后根据当前用户角色和 `access_scope` 决定默认进入位置：仅 `PORTAL` 的普通用户进入前台；拥有 `ADMIN` 或 `BOTH` 的管理员、商家、版主可进入对应后台。双端用户可在右上角角色或工作台切换入口中切换当前访问视图，例如商家可在前台和商家后台之间切换。

### 4.8 配置项管理

配置项管理作为基础模块，由超级管理员维护。建议先支持：

```text
user_org_relation_mode = ONE_TO_ONE / ONE_TO_MANY
anonymous_portal_access = true / false
guest_admin_access = false
```

其中 `user_org_relation_mode` 默认 `ONE_TO_MANY`，表示一个用户可以加入多个组织；如果后台配置为 `ONE_TO_ONE`，则用户只能加入一个组织。当前代码仍保留 `role_level_enabled` 内置配置和校验，用于兼容已有配置项；后续配置项体验优化时，应把它解释为“是否启用角色层级管理约束”，而不是固定某一套层级模型。

## 五、数据库设计草案

字段命名建议使用下划线风格，实体类使用驼峰映射。通用字段建议统一：

```text
id, create_by, create_time, update_by, update_time, deleted
```

### 5.1 系统权限表

| 表名 | 用途 |
| :--- | :--- |
| `sys_user` | 用户账号、基础资料、状态 |
| `sys_role` | 角色信息 |
| `sys_menu` | 菜单、目录、页面、iframe、外链 |
| `sys_permission` | 按钮或接口权限标识 |
| `sys_org` | 组织、部门、社团、分组树 |
| `sys_config` | 系统配置项 |
| `sys_user_role` | 用户角色关联 |
| `sys_user_org` | 用户组织关联 |
| `sys_role_menu` | 角色菜单关联 |
| `sys_role_permission` | 角色按钮权限关联 |
| `sys_role_org` | 角色自定义数据权限组织关联 |

关键约束：

- `sys_user.user_name` 唯一。
- `sys_role.role_code` 唯一，例如 `R_SUPER`、`R_ADMIN`。
- `sys_menu.name` 建议唯一，对应前端路由 name。
- `sys_permission.permission_code` 建议按模块命名，例如 `system:user:add`。
- `sys_org` 使用树形结构，支持部门、社团、分组等通用组织模型。
- 用户和组织默认是一对多关系，通过 `sys_user_org` 维护；是否限制为一对一由 `sys_config` 控制。
- 所有删除优先使用逻辑删除。

### 5.2 组织树与数据权限表

组织统一命名为 `org`，用于覆盖部门、社团、分组等业务场景。

`sys_org` 建议字段：

```text
id, parent_id, ancestors, org_name, org_code, org_type, sort,
enabled, create_by, create_time, update_by, update_time, deleted
```

角色数据权限建议在 `sys_role` 保存范围枚举：

```text
ALL                 全部数据
SELF                本人数据
CURRENT_ORG         当前组织数据
CURRENT_ORG_AND_SUB 当前组织及下级组织数据
CUSTOM_ORG          自定义组织数据
```

当 `data_scope=CUSTOM_ORG` 时，通过 `sys_role_org` 保存可访问组织范围。

组织树查询应用当前登录用户的数据权限：

- 超级管理员 `R_SUPER` 或角色数据范围包含 `ALL` 时返回全量组织树。
- `CURRENT_ORG` 返回当前用户直接关联组织。
- `CURRENT_ORG_AND_SUB` 返回当前用户直接关联组织及其所有下级组织。
- `CUSTOM_ORG` 返回角色自定义组织及其所有下级组织。
- `SELF` 在组织树场景按当前用户直接关联组织处理，避免返回全量组织。
- 用户拥有多个角色时按最大可见范围合并，并保留可见节点的祖先节点，保证树结构可正常挂载。

### 5.3 文章业务表

| 表名 | 用途 |
| :--- | :--- |
| `article_category` | 文章分类 |
| `article` | 文章主表 |
| `article_attachment` | 文章附件关系 |
| `article_comment` | 评论或留言 |

文章主表建议包含：

```text
id, title, category_id, cover_url, summary, content_html, content_text,
visible, status, view_count, comment_count, publish_time, create_by, create_time, update_by, update_time, deleted
```

当前文章内容形态先固定为：分类、标题、封面、摘要、富文本正文、状态、作者、发布时间、浏览数、评论数和附件。文章状态使用 `DRAFT`、`PUBLISHED`、`OFFLINE`。附件通过 `article_attachment` 关联 `file_resource`，一篇文章可夹带多个附件。评论数使用 `article.comment_count` 冗余字段维护，一级评论和所有回复都计入评论数，新增正常评论时递增，管理删除评论时按删除策略递减。删除文章时先逻辑删除文章，附件资源不物理删除，避免同一文件被其他业务复用时丢失。

文章评论表 `article_comment` 建议字段：

```text
id, article_id, parent_id, root_id, content, status,
user_id, user_name, user_avatar, ip_address, user_agent,
create_time, update_time, deleted
```

评论设计确认：

- 评论定位为文章详情底部评论区，参考 CSDN 类文章详情模式，不再按独立留言墙作为主流程设计。
- 评论读取接口可匿名访问，但读取结果必须跟随文章可访问性；游客只能读取已发布且可见文章的评论，具备文章管理权限或超级管理员身份的用户可按后台规则读取。
- 评论使用一级评论 + 子回复结构，`parent_id=0` 表示一级评论，`root_id` 指向所属一级评论。业务允许回复任意一条未删除评论；回复子回复时 `parent_id` 指向被回复评论，`root_id` 仍指向一级评论，前端保持两层展示并显示“回复某人”。
- 评论状态使用 `NORMAL`、`HIDDEN`、`DELETED`。管理端删除采用状态 `DELETED`，隐藏当前节点内容并保留子回复，避免评论树断裂；数据库 `deleted` 字段仅保留给未来批量清理或后台维护使用，不参与本轮管理删除。`DELETED` 状态不可恢复为 `NORMAL/HIDDEN`，避免恢复后只剩占位内容。
- 评论内容按纯文本处理，不支持富 HTML；前端使用文本插值展示，后端保存前做空内容、500 字长度上限和控制字符校验。
- 评论数量由 `article.comment_count` 冗余维护：一级评论和所有回复都计入数量，新增正常评论递增，隐藏评论仍计入数量，删除评论不计入数量；计数更新使用数据库原子增减，避免并发读改写丢失。
- 挂载在已删除评论下的历史子回复继续保留记录并可在评论树中展示，用于保留讨论上下文；但已删除评论本身不允许继续新增回复。
- 一级评论下回复数量超过 3 条时，前端默认折叠展示前 3 条，并提供展开全部和收起功能，交互参考哔哩哔哩评论区。
- 评论隐藏功能由系统配置 `article_comment_hide_enabled` 控制，默认开启且仅超级管理员可配置。超级管理员始终可隐藏、恢复和删除；有 `article:comment:manage` 权限的普通管理员仅在配置开启时可隐藏、恢复和删除；文章发布者仅在配置开启时可隐藏或恢复自己文章下的评论，不能删除；评论作者只能删除自己的评论，不能隐藏或恢复。
- 公开评论响应不返回内部 `userId`，只返回昵称、头像、`mine` 布尔值和后端计算出的 `canHide/canRestore/canDelete` 操作标志。隐藏和删除评论内容必须在后端脱敏，分别返回“该评论已隐藏”和“该评论已删除”。

## 七、后端安全设计

### 7.1 密码安全

- 使用 BCrypt 或 Argon2 存储密码哈希。
- 不存储明文密码。
- 当前登录用户修改密码必须校验旧密码，新密码不能与当前密码相同，修改成功后前端强制重新登录。
- 模板/开发环境初始化管理员为 `admin/admin123`，仅用于本地初始化和 Mock SQL；上线前必须修改。
- 登录失败建议增加频率限制或账号临时锁定。

### 7.2 接口鉴权

- 所有 `/api/**` 默认需要认证，登录和前台公开只读接口除外。
- 用户、角色、菜单、文章发布、上传等接口必须校验权限。
- 文章评论列表属于前台公开只读接口，访问规则跟随文章详情：能打开文章即可查看评论。
- 评论新增和回复要求登录，游客只能查看公开评论；评论隐藏、恢复和删除由评论治理规则控制，后端返回当前登录用户对每条评论的 `canHide/canRestore/canDelete` 操作标志。
- 前端按钮隐藏只改善体验，不能替代后端权限判断。
- 游客只能访问前台公开只读接口，不能进入后台，不能执行写操作。

### 7.3 XSS 与富文本

- 文章入库前过滤危险 HTML。
- 评论按纯文本保存和展示，不支持富 HTML；后端保存前做空内容、长度和控制字符校验。
- 返回富文本前再次净化。
- 禁止 `script`、事件属性、`javascript:` URL。
- SVG 上传默认禁止，除非有严格白名单和净化流程。

### 7.4 数据权限

- Service 层处理业务查询时必须应用数据权限条件。
- Controller 不直接拼接数据权限条件。
- `ALL` 只能授予可信高权限角色。
- `CUSTOM_ORG` 通过角色和组织关联表控制可见组织范围。
- 用户同时拥有多个角色时，数据范围按最大可见范围合并。
- 组织树查询由 Service 层计算可见组织 ID，Controller 只传递当前登录用户身份并返回结果。

### 7.5 CORS 与 CSRF

- 开发环境允许 `http://localhost:*`。
- 生产环境只允许正式前端域名。
- 如果采用 Cookie 认证，必须启用 CSRF 防护。
- 如果采用 Bearer Token，仍需限制 CORS 来源。

### 7.6 iframe 白名单

菜单保存和菜单返回时都要校验 iframe 地址：

- 只允许 `https://`。
- 只允许配置中的域名白名单。
- 不允许 `javascript:`、`data:`、`file:` 等协议。

### 7.7 审计日志

建议记录：

- 登录成功、登录失败、登出。
- 用户、角色、菜单、文章、上传等关键操作。
- 组织、配置项、角色层级、数据权限等基础配置变更。
- 操作人、IP、User-Agent、请求路径、业务对象 ID、结果、失败原因。

敏感字段如密码、token、Cookie 不得写入日志。

接口约定：

| 接口 | 方法 | 说明 |

业务规则：

- 用户和组织默认是一对多关系，由 `sys_config.user_org_relation_mode=ONE_TO_MANY` 控制。

- 保存用户组织关系时，后端会校验组织 ID 是否存在，避免关系表出现脏数据。
- `sys_user_org.primary_org=1` 表示主组织，多组织场景下第一个组织作为主组织。

- `mock_001_system.sql` 已补充 `moderator` 演示用户。
- `moderator` 绑定 `R_MODERATOR` 角色。
- `moderator` 绑定内容管理组和运营管理组，用于验证一人多组织和角色数据权限。
## 十五、本轮设计：认证收尾与评论反垃圾策略

更新时间：2026-06-02

### 15.1 认证模块收尾范围

本轮认证模块只收尾当前已经进入工作区的注册和忘记密码改动，不新增生产级验证码、邮箱、短信或审计链路。

收尾内容：

- 保持 `POST /api/auth/register` 作为开发期普通用户注册接口，注册成功后创建普通用户，绑定默认角色 `R_USER` 和默认组织 `ORG_TEMPLATE`。
- 保持 `POST /api/auth/reset-password` 作为开发期账号密码重置接口，只校验账号存在、新密码与当前密码不同。
- 前端注册页和忘记密码页调用真实接口，错误信息走现有请求封装和页面提示。
- 后端保持 Controller 只处理参数和响应，注册、重置密码业务继续放在 `AuthService`。
- 本轮不补充自动化测试；完成后至少执行后端编译和前端类型检查。

生产安全边界：

- 自助注册上线前必须明确是否开放；如果不开放，生产菜单或入口需要隐藏。
- 忘记密码上线前必须增加验证码、邮箱或短信校验、频率限制和审计日志；当前开发期接口不得作为生产找回密码能力直接暴露。

### 15.2 评论反垃圾策略

本轮评论反垃圾采用轻量同步校验，不引入 Redis，优先降低刷屏、重复提交和敏感词命中风险。敏感词规则已从临时常量升级为数据库敏感词库，由系统管理统一维护。

后端规则：

- 发布评论或回复前，仍先完成文章存在性、内容归一化、父评论合法性校验。
- 增加同一用户短时间发布频率限制：同一用户在 60 秒内最多发布 5 条评论或回复。
- 增加同一用户同一文章重复内容限制：同一用户在同一文章下 10 分钟内提交完全相同的归一化内容时拒绝。
- 增加敏感词拒绝策略：读取启用状态的敏感词库规则，命中后直接返回业务错误和命中词列表；后续可继续升级为审核队列或独立内容安全服务。
- 上述校验全部放在 `ArticleCommentService`，Controller 不承载真实业务判断。

实现方式：

- 优先使用 MyBatis-Plus `LambdaQueryWrapper` 查询 `article_comment`，避免自定义 SQL。
- 频率限制通过查询当前用户在 `create_time >= now - 60s` 范围内的评论数量实现。
- 重复内容限制通过查询当前用户、当前文章、同一内容、`create_time >= now - 10min` 的评论记录实现。
- 敏感词使用 `comment_sensitive_word` 表作为底层存储，通过系统管理“敏感词库”维护，新增、编辑、启停和删除后即时影响后续文本写入校验。

限制说明：

- 当前策略基于数据库查询，适合模板开发阶段；高并发生产环境应升级为 Redis 滑动窗口或令牌桶。
- 当前未按 IP、设备指纹或用户代理限流；游客不能发评论，因此本轮先按登录用户维度处理。
- 当前敏感词库只是基础拦截，不替代完整审核系统。

### 15.3 认证验证码补充设计

本轮在注册、忘记密码和当前用户修改密码三个入口增加 4 位验证码。验证码只用于降低自动化提交风险，不替代忘记密码生产级邮箱、短信或身份校验。

验证码规则：

- 长度固定 4 位。
- 字符范围为数字和英文字母：`0-9`、`A-Z`、`a-z`。
- 校验不区分大小写。
- 验证码以图片形式展示，不直接返回明文。
- 图片需要包含干扰线、噪点、字符随机偏移和轻微旋转，避免页面直接明文展示。
- 不使用 Redis；本轮采用后端内存缓存，带过期时间和一次性消费。

接口设计：

```text
GET /api/auth/captcha
```

返回：

```ts
{
  captchaId: string
  imageBase64: string
  expiresIn: number
}
```

请求调整：

- `POST /api/auth/register` 增加 `captchaId`、`captchaCode`。
- `POST /api/auth/reset-password` 增加 `captchaId`、`captchaCode`。
- `PUT /api/user/profile/password` 增加 `captchaId`、`captchaCode`。

后端实现：

- 新增 `CaptchaService`，负责生成、存储、校验和消费验证码。
- 存储使用 `ConcurrentHashMap<String, CaptchaEntry>`，记录验证码小写值、过期时间和创建时间。
- 默认有效期 5 分钟；校验成功后立即删除，校验失败不删除，过期自动清理。
- 采用定时清理或每次生成/校验时顺带清理过期记录，避免长期堆积。
- Controller 只处理接口输入输出；注册、重置密码、修改密码仍由 Service 处理真实业务。

限制说明：

- 内存验证码在服务重启后全部失效。
- 单机开发和单体部署可用；多实例部署时需要升级为 Redis 或共享存储。
- 忘记密码即使增加验证码，仍不能直接作为生产安全找回流程；生产前必须增加邮箱、短信、一次性链接或人工审核。

### 15.4 全站敏感词管理与内容写入集成设计

敏感词能力已从评论专用升级为系统级内容安全能力。当前保留 `comment_sensitive_word` 表作为底层存储，页面和权限语义统一为“敏感词”，并覆盖文章、评论、用户、角色、组织、配置项、菜单和注册等文本写入口。

后端数据模型：

```text
comment_sensitive_word
id, word, match_type, enabled, remark, create_time, update_time, create_by, update_by, deleted
```

字段说明：

- `word`：敏感词内容，不能为空，建议最长 100 字。
- `match_type`：匹配方式，当前支持 `CONTAINS`，后续可扩展 `REGEX`。
- `enabled`：是否启用，1 启用，0 禁用。
- `deleted`：逻辑删除字段。

接口设计：

```text
GET    /api/system/sensitive-words
POST   /api/system/sensitive-words
PUT    /api/system/sensitive-words/{id}
PATCH  /api/system/sensitive-words/{id}/status
DELETE /api/system/sensitive-words/{id}
```

兼容说明：旧路径 `/api/article/comment/sensitive-words` 暂时保留，便于旧前端或脚本过渡；新代码统一使用 `/api/system/sensitive-words`。

权限设计：

- 新增权限码：`system:sensitive-word`。
- 仅超级管理员或拥有该权限码的管理员可管理敏感词。
- 初始 Mock SQL 中只给 `R_SUPER` 绑定该权限；如需评论管理员可维护，后续再明确授权。

内容写入集成：

- 后端新增通用 `SensitiveWordGuard`，业务 Service 在保存前按字段调用校验。
- `POST /api/article/comment` 在保存前读取启用敏感词；命中时不保存评论，不增加 `article.comment_count`。
- 文章新增和编辑校验标题、摘要和正文纯文本。
- 用户新增、编辑和个人资料更新校验用户名、昵称、手机号、邮箱等文本字段。
- 角色、组织、配置项、菜单的新增和编辑校验名称、编码、描述、标题、权限码等用户可输入文本字段。
- 注册校验用户名。
- 命中敏感词时直接返回业务错误，并在错误扩展信息中返回命中的敏感词列表。
- 当前按不区分大小写的包含匹配处理。
- 为避免每次评论都全表扫描，Service 可使用本地短 TTL 缓存启用敏感词列表；新增、编辑、启停、删除敏感词后清空缓存。

前端评论发布交互：

- 用户输入评论内容后点击发布。
- 后端先做敏感词校验，命中敏感词时拒绝保存。
- 前端收到敏感词错误后不得清空输入框，必须保留用户当前输入内容，便于用户修改。
- 前端在评论输入区附近提示当前命中的敏感词，例如“内容包含敏感词：xxx、yyy，请修改后再发布”。
- 敏感词校验通过后，后端再保存评论并递增评论数，前端清空输入框并刷新评论列表。

评论发布流程：

```text
用户输入 -> 点击发布 -> 后端校验
  -> 命中敏感词：拒绝保存 + 返回命中词列表 + 前端保留输入并提示
  -> 通过校验：保存评论 + 评论数加 1 + 前端清空输入并刷新列表
```

前端集成：

- 在现有文章评论管理或评论演示页改造中加入敏感词管理入口。
- 入口仅超级管理员可见。
- 页面支持分页、搜索、添加、编辑、启用/禁用和删除。

本轮取舍：

- 暂不做审核队列。
- 暂不做正则匹配启用，避免误配造成性能或误杀问题。
- 暂不补自动化测试，但完成后至少执行后端编译和前端类型检查。
