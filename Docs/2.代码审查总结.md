# 代码审查总结

> 本文档基于 `Code/art-design-pro` 当前代码和 `Docs/1.项目说明文档.md` 复核整理。目标是形成可信、问题明确、适合上线前整改的审查清单。

最后更新：2026-06-01

## 一、审查结论

当前项目适合作为后台管理系统前端模板继续开发，但还不能直接作为线上生产项目发布。系统管理页已逐步接入真实后端接口，认证请求头已统一为 Bearer JWT，后端系统管理主要接口已接入权限码校验并补充部分权限测试。剩余上线阻断主要集中在演示账号和演示页面、未净化 HTML 渲染、前端可读 token 持久化、iframe 外链缺少约束、上传和富文本安全边界不完整等问题。

本审查按可信度和上线影响重新分级：

- P0：上线阻断，必须修复或明确替代方案。
- P1：上线前应修复，否则存在安全、稳定性或后端对接风险。
- P2：建议修复，主要影响维护性、体验或长期质量。
- 降级/误判：原记录有代码依据，但风险描述不准确或不应作为高优先级问题。

## 二、P0 上线阻断项

### 2.1 多处 HTML 直接渲染，存在 XSS 风险

| 项 | 内容 |
| :--- | :--- |
| 证据 | `src/views/article/detail/index.vue` 使用 `v-html="articleHtml"`；`src/components/core/text-effect/art-text-scroll/index.vue` 使用 `v-html="text"`；`src/components/core/theme/theme-svg/index.vue` 使用 `v-html="svgContent"`；`src/views/widgets/wang-editor/index.vue` 渲染富文本预览；`src/utils/sys/upgrade.ts` 使用 `dangerouslyUseHTMLString`。 |
| 风险 | 后端内容、外部文章、富文本、SVG 或升级日志被污染时，可执行恶意脚本并窃取用户状态。 |
| 可信度 | 高。代码中存在直接证据。 |
| 修复建议 | 引入 DOMPurify 或同等 HTML 消毒方案；封装 `sanitizeHtml()`；所有 `v-html` 前必须净化；SVG 只允许可信本地资源或严格白名单；升级通知避免 `dangerouslyUseHTMLString`。 |
| 后端联动 | 后端保存富文本前也应做内容净化或白名单过滤，至少过滤脚本、事件属性、危险 URL 协议。 |

### 2.2 token 和刷新 token 持久化到 localStorage

| 项 | 内容 |
| :--- | :--- |
| 证据 | `src/store/modules/user.ts` 定义 `accessToken`、`refreshToken`，并通过 `pinia-plugin-persistedstate` 使用 `localStorage` 持久化。 |
| 风险 | 一旦出现 XSS，攻击者可直接读取 token 并冒用账号。 |
| 可信度 | 高。代码中存在直接证据。 |
| 修复建议 | 上线推荐使用 HttpOnly + Secure + SameSite Cookie；如短期继续使用 token，建议 accessToken 内存化、refreshToken 不暴露给 JS、缩短有效期并补充刷新和失效机制。 |
| 后端联动 | 后端需决定认证方式。Cookie 方案要支持 CSRF 防护、SameSite、CORS credentials；Bearer 方案要提供过期、刷新、登出失效策略。 |

### 2.3 登录页包含演示账号密码

| 项 | 内容 |
| :--- | :--- |
| 证据 | `src/views/auth/login/index.vue` 中 `accounts` 包含快捷选择账号；开发管理员为 `admin/admin123`，其他演示账号仍使用 `123456`。 |
| 风险 | 上线后暴露演示账号和登录入口，容易被误用或攻击。 |
| 可信度 | 高。代码中存在直接证据。 |
| 修复建议 | 演示账号只在开发环境启用；生产构建不包含账号列表、默认密码和自动填充逻辑；后端禁止任何默认弱密码账号上线。 |
| 后端联动 | 后端初始化数据不得保留默认口令；首次登录应强制修改密码或使用管理员创建流程。 |

### 2.4 外部数据依赖仍未完全本地化

| 项 | 内容 |
| :--- | :--- |
| 证据 | 当前已接入 `mock/local-mock.ts` 覆盖开发期登录、用户信息、用户列表、角色列表、菜单树、文章分类和文章详情；系统管理用户、角色、菜单、组织、配置项页面已接入真实后端接口；文章评论、升级日志、部分组件示例和看板页仍存在 `src/mock/temp/*` 或静态示例数据。 |
| 风险 | 外部服务不稳定、数据不可控、无法离线调试，且返回内容被污染时会影响前端安全。 |
| 可信度 | 高。代码和文档均有证据。 |
| 当前进展 | 已通过 Vite middleware 建立本地 Mock；系统管理 CRUD 已逐步切换到真实后端；文章分类、文章详情仍先使用本地 Mock 兜底。 |
| 修复建议 | 继续将文章发布、文章列表、评论、上传等演示数据迁移到真实后端；生产环境禁止依赖外部 Mock 或静态演示数据。 |

### 2.5 iframe 外链缺少白名单和沙箱限制

| 项 | 内容 |
| :--- | :--- |
| 证据 | `src/views/outside/Iframe.vue` 将菜单中的 `meta.link` 直接赋给 iframe `src`；`IframeRouteManager` 将 iframe 路由保存到 `sessionStorage`。 |
| 风险 | 菜单数据或本地存储被污染时，可能嵌入任意外部站点，造成钓鱼、点击劫持或敏感上下文暴露。 |
| 可信度 | 高。代码中存在直接证据。 |
| 修复建议 | 对 `meta.link` 做协议和域名白名单；iframe 增加 `sandbox`；只允许 HTTPS；后端返回菜单时也校验 iframe 地址。 |

## 三、P1 上线前应修复项

### 3.1 后端接口权限测试仍需继续补齐

当前前端请求封装已统一发送 `Authorization: Bearer <token>`，后端也按 Bearer JWT 解析；该旧风险已过期。系统管理用户、角色、菜单、组织、配置项 Controller 已补齐 MockMvc 权限入口测试，覆盖无 token 401、无权限 403、有权限 200 和主要写接口权限校验；无效或过期 JWT 已统一返回 HTTP 401。剩余工作是补真实角色账号页面联调和长期 token 失效机制。

### 3.2 注册、文章发布等流程仍是演示逻辑

`src/views/auth/register/index.vue` 使用模拟成功逻辑；文章发布仍存在 TODO 和仅 `console.log` 的保存逻辑；系统管理用户、角色、菜单、组织、配置项页面已接入真实后端接口。上线前需要后端补齐注册、文章和上传等真实接口，前端移除模拟成功逻辑。

### 3.3 WebSocket 环境变量仍使用 Vue CLI 风格

`src/utils/socket/index.ts` 使用 `process.env.VUE_APP_LOGIN_WEBSOCKET`。Vite 项目应改为 `import.meta.env.VITE_WS_URL`，并增加 URL 校验。

### 3.4 WebSocket 定时器和连接切换逻辑不稳定

心跳用 `setInterval` 创建，但统一用 `clearTimeout` 清理；URL 变化时调用 `init()`，已有 OPEN 连接会直接 return。建议按 timer 类型分别清理，URL 变化时关闭旧连接再建立新连接，并提供销毁实例能力。

### 3.5 富文本和上传接口安全边界不完整

wangEditor 上传头直接携带当前 token，上传 URL 来自环境变量或 props。上线前必须要求上传接口使用 HTTPS，后端校验 MIME、后缀、大小，图片重编码，返回可信 URL，富文本内容保存前净化。

### 3.6 响应拦截器对异常响应不够稳健

`src/utils/http/index.ts` 默认从 `response.data` 解构 `code`、`msg`。如果后端返回空 body、字符串、非标准 JSON，可能抛出非预期错误。建议解构前判断响应结构，并对文件流、纯文本、204 响应提供单独处理。

### 3.7 `@vue/compiler-sfc` 与 Vue 版本不一致

`vue` 为 `^3.5.21`，`@vue/compiler-sfc` 为 `^3.0.5`。建议升级并锁定到与 Vue 运行时兼容的版本。

### 3.8 Excel 依赖存在安全风险

`xlsx ^0.18.5` 被多个安全数据库标记存在 Prototype Pollution / ReDoS 风险。若解析不可信文件，应限制上传大小和类型，解析放入 Worker，并评估替代库或安全版本。

### 3.9 权限指令直接移除 DOM

`src/directives/core/auth.ts` 和 `src/directives/core/roles.ts` 无权限时调用 `removeChild`。权限或角色变化后，被移除元素无法恢复。建议优先使用 `v-if="hasAuth(...)"`，或把指令改为响应式显示控制。

## 四、P2 建议修复项

### 4.1 中文注释和日志存在乱码

多个文件在终端读取时出现乱码，影响维护、审查和日志定位。建议统一 UTF-8 编码，配置编辑器、终端和 Git 属性，并逐步修复乱码注释。

### 4.2 `no-explicit-any` 被关闭

`eslint.config.mjs` 中 `@typescript-eslint/no-explicit-any` 为 `off`。建议先改为 `warn`，逐步收敛核心模块中的 `any`。

### 4.3 生产环境 console 管理不完整

多个页面和工具中存在 `console.log`，部分日志可能包含表单、富文本、文件等数据。建议业务日志加 `import.meta.env.DEV`，构建时移除非必要 console。

### 4.4 部分页面直接依赖本地静态 Mock

部分页面直接 import `src/mock/temp/*`，例如评论组件、文章评论页和升级日志。系统管理主流程已改为真实 API。建议继续把剩余演示页改为开发环境 API 层，生产构建不引入 Mock 数据。

### 4.5 路由和 Store 存在循环依赖倾向

`userStore` 引入 router，route guard 又使用 userStore。当前可运行，但重构或测试时初始化顺序脆弱。建议将路由重置、登出跳转等副作用抽到独立服务或调用层注入。

### 4.6 CSS 单位问题

`LoginLeftView.vue` 中 `animation-delay: 0.5` 缺少 `s`，动画延迟不生效。应改为 `0.5s`。

## 五、原审查中的降级或误判项

### 5.1 `App.vue` 缺少 `storeToRefs` 导入

当前 Vite 配置了 `unplugin-auto-import`，自动导入包含 `pinia`，因此不应直接判定为启动崩溃。建议显式 import 提升可读性，但不属于 P0。

### 5.2 `redirect` 开放重定向

Vue Router 的 `router.push(redirect)` 通常处理站内路由，不等同于 `window.location.href`。但仍建议限制 `redirect` 只能是站内路径，避免异常路由和历史记录污染。

### 5.3 `Authorization` 必须带 Bearer

Bearer 是常见标准方案，但前后端可以约定裸 token。该项应作为接口契约问题处理，而不是绝对漏洞。

### 5.4 `__dirname` 在 Vite ESM 中必然失效

`vite.config.ts` 中使用 `__dirname` 不理想，但是否失效取决于 Vite 执行环境和转译处理。建议改为 `fileURLToPath(new URL(..., import.meta.url))`，但不应列为高危。

### 5.5 锁屏加密

前端锁屏密钥在构建产物中可见，不能作为真正安全边界。该功能最多是 UI 层防误触，不应用于保护敏感数据。

## 六、推荐整改顺序

1. 移除生产演示账号和外部 Mock 依赖。
2. 完成 HTML 净化、富文本安全和 iframe 白名单。
3. 明确认证方案，优先切换 HttpOnly Cookie 或至少改造 token 生命周期。
4. 补真实角色账号页面联调，验证菜单、按钮和接口 403 三者一致。
5. 补齐文章、上传、注册等真实后端接口契约。
6. 修复 WebSocket、上传、响应拦截器等稳定性问题。
7. 统一依赖版本、编码、console 和类型规则。
8. 逐步替换剩余页面中的本地静态数据和 TODO 逻辑。
