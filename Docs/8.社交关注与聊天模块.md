# 社交关注与聊天模块

最后更新：2026-06-03

## 一、模块范围

本模块是当前正在开发的业务模块，目标是实现：

- 关注。
- 粉丝。
- 互关。
- 拉黑。
- 私信会话。
- 文本消息。
- 未读数和已读状态。
- 顶部栏聊天入口。

本轮做完整后端业务。前端页面可以参考基准项目 `Code/art-design-pro-main/src/views/template/chat/index.vue` 的聊天页面布局，但需要接入真实业务接口。

## 二、界面默认配置

当前项目默认配置调整为：

| 配置项 | 默认值 | 说明 |
| :--- | :--- | :--- |
| `menuType` | `MenuTypeEnum.DUAL_MENU` | 默认双列菜单 |
| `showFastEnter` | `false` | 默认不显示快速入口 |
| `pageTransition` | `fade` | 默认淡入淡出 |

顶部栏聊天按钮：

- 点击跳转 `/social/chat`。
- 不打开右侧聊天弹窗。
- `/social/chat` 是隐藏路由，不注册到侧边栏菜单。

## 三、用户来源

- 前台和后台用户本轮统一复用 `sys_user`。
- 只有启用且未删除用户可以参与关注和聊天。
- 用户不能关注、拉黑或私信自己。

## 四、关注和粉丝规则

- A 关注 B：A 是 B 的粉丝。
- 互关：A 关注 B 且 B 关注 A。
- 取消关注只删除单向关注关系。
- 重复关注按幂等成功处理，不重复插入有效关注关系。
- 已拉黑或被对方拉黑时不能关注。

## 五、拉黑规则

- A 拉黑 B 后，B 不能给 A 发私信。
- A 拉黑 B 后，B 不能关注 A。
- 拉黑时自动取消双方关注关系。
- 解除拉黑后不自动恢复关注。
- 拉黑列表只对当前用户自己可见。

## 六、私信规则

本轮采用普通 HTTP 拉取和发送消息，不实现 WebSocket。

私信额度：

- 任意已登录用户可以给未关注用户发送私信。
- 非互关关系下，A 给 B 最多连续发送 3 条。
- B 回复 1 条后，A 对 B 的额度恢复为 3 条。
- B 不回复时，A 第 4 条消息拒绝发送。
- 双方互关后不受 3 条额度限制。
- 取消互关后重新回到非互关额度规则。

消息内容：

- 本轮只实现文本消息。
- 文本不能为空。
- 文本最多 1000 字。
- 发送前接入 `SensitiveWordGuard`。
- 图片、附件、语音和撤回暂不实现。

## 七、数据模型

| 表 | 说明 |
| :--- | :--- |
| `social_follow` | 关注关系 |
| `social_block` | 拉黑关系 |
| `social_conversation` | 双人会话 |
| `social_message` | 聊天消息 |

### 7.1 social_follow

```text
id, follower_id, following_id, create_time, deleted
```

唯一约束：

```text
(follower_id, following_id, deleted)
```

### 7.2 social_block

```text
id, blocker_id, blocked_id, create_time, deleted
```

唯一约束：

```text
(blocker_id, blocked_id, deleted)
```

### 7.3 social_conversation

```text
id, user_a_id, user_b_id, last_message_id, last_message_time, create_time, update_time, deleted
```

约定：

- `user_a_id` 和 `user_b_id` 按用户 ID 从小到大保存。
- 两个用户之间只有一个有效会话。

### 7.4 social_message

```text
id, conversation_id, sender_id, receiver_id, content, message_type, read_time, create_time, deleted
```

约定：

- `message_type` 本轮固定为 `TEXT`。
- `read_time` 为空表示未读。

## 八、接口

用户：

| 接口 | 方法 | 说明 |
| :--- | :--- | :--- |
| `/api/social/users` | GET | 搜索用户 |
| `/api/social/users/{id}` | GET | 用户社交资料 |

关注和拉黑：

| 接口 | 方法 | 说明 |
| :--- | :--- | :--- |
| `/api/social/following` | GET | 我的关注 |
| `/api/social/followers` | GET | 我的粉丝 |
| `/api/social/follow/{targetUserId}` | POST | 关注 |
| `/api/social/follow/{targetUserId}` | DELETE | 取消关注 |
| `/api/social/blocks` | GET | 拉黑列表 |
| `/api/social/block/{targetUserId}` | POST | 拉黑 |
| `/api/social/block/{targetUserId}` | DELETE | 解除拉黑 |

会话和消息：

| 接口 | 方法 | 说明 |
| :--- | :--- | :--- |
| `/api/social/conversations` | GET | 会话列表 |
| `/api/social/conversations/{conversationId}/messages` | GET | 消息列表 |
| `/api/social/messages` | POST | 发送消息 |
| `/api/social/conversations/{conversationId}/read` | POST | 标记已读 |
| `/api/social/unread-count` | GET | 未读总数 |

发送消息请求：

```ts
interface SendSocialMessageRequest {
  receiverId: number
  content: string
}
```

额度不足响应：

```json
{
  "code": 400,
  "msg": "对方回复前最多发送3条私信",
  "data": {
    "remainingQuota": 0
  }
}
```

## 九、前端页面

页面路径：

```text
src/views/social/chat/index.vue
```

页面结构：

- 左侧：会话、关注、粉丝、拉黑标签。
- 中间：消息列表、输入区和当前会话操作。
- 移动端：列表和消息区纵向切换。
- 页面挂载在主布局隐藏路由 `/social/chat` 下，侧边栏不显示该路由入口，顶部栏聊天按钮负责进入该页面。
- 页面高度以主布局内容区剩余高度为基准，消息列表只在聊天主体内部滚动，不能因消息增多或长文本撑大整页。
- 双列菜单模式下需要优先保证聊天主体宽度；当前已取消最右侧资料卡，聊天页只保留会话列表和主聊天区，以便给双列菜单预留更多横向空间。

交互：

- 非互关显示剩余额度。
- 额度为 0 时禁用发送按钮。
- 敏感词错误保留输入内容。
- 打开会话后标记对方消息已读。

## 十、WebSocket 后续规划

本轮不实现 WebSocket。后续升级时：

- HTTP 发送接口仍是权威入口。
- WebSocket 只负责推送新消息、未读数和在线状态。
- 端点可规划为 `/ws/social`。
- WebSocket 鉴权必须基于 JWT，不能信任前端传入用户 ID。
- 需要先修复现有 `src/utils/socket/index.ts` 的 Vite 环境变量、重连和销毁逻辑。

## 十四、本次路由与接口修复记录

更新时间：2026-06-03

- `/social/chat` 已调整为主布局下的隐藏路由：父级使用 `@views/index/index.vue`，子路由 `SocialChat` 使用 `meta.isHide = true`，因此不显示在侧边栏，但页面仍在后台主框架内渲染。

- `/api/social/conversations` 500 根因定位为 `SocialStructMapper` 放在 `com.template.social.mapper` 包内，被 MyBatis `@MapperScan` 扫描成 MyBatis Mapper 代理，运行时调用 MapStruct 方法时触发 `Invalid bound statement`。
- 修复方式：将 MapStruct 转换器移动到 `com.template.social.convert.SocialStructMapper`，并在消息转换方法上显式声明 `senderName`、`senderAvatar`、`mine`、`read`、`createTime` 映射，避免格式化时间参数被 MapStruct 忽略。

## 十五、本次聊天页双列菜单适配记录

更新时间：2026-06-04

- 问题定位：聊天页根容器使用 `min-height`，内部消息列表虽设置 `flex: 1`，但父级没有稳定高度，输入对话和消息增长时可能撑大整页。
- 问题定位：聊天页固定三列 `320px + 1fr + 260px`，未考虑外层双列菜单已占用宽度，双列菜单展开时聊天主体容易被压缩。
- 修复方案：根容器改为使用主布局剩余高度，三列宽度改为 `clamp` 响应式宽度，并提前在中等宽度隐藏右侧资料栏。
- 修复方案：小宽度切换为纵向布局，左侧列表与聊天主体分别在自身区域滚动，消息气泡增加长文本换行约束。
- 问题定位：`/social/chat` 是隐藏静态路由，不在双列菜单一级菜单树内，导致侧边栏无选中态；切换到其他二级路由前必须点击一级菜单，而旧逻辑会立即跳转到该一级菜单下第一个叶子路由。
- 修复方案：双列菜单增加当前一级菜单选择状态；一级菜单点击只切换右侧二级菜单，不立即跳转，二级菜单点击才进入页面；隐藏路由保留上一次可见一级菜单选中态。
- 修复方案：双列菜单第二列增加两个收起入口：侧边栏边缘收起按钮常驻显示，顶部栏在双列模式下显示专用收起/展开按钮；两者共用 `menuOpen` 状态，主内容区随菜单宽度自动适配。
- 修复方案：聊天页左侧分类选项卡增加独立内边距容器，保留四项等宽分布，同时让首尾分类和搜索框、会话列表的边距视觉对齐。
- 验证方式：前端类型检查和浏览器聊天页回归，重点验证双列菜单展开、折叠、发送长文本、消息增多后的内部滚动。

## 十六、本次聊天体验与附件能力设计

更新时间：2026-06-04

实施范围：

- 消息展示改为接近微信/QQ 的左右气泡样式，自己消息在右侧，对方消息在左侧。
- 文本消息继续使用 `TEXT` 类型，图片消息新增 `IMAGE` 类型，附件消息新增 `FILE` 类型。
- 暂时只做 emoji 表情，不做图片表情包、收藏表情或表情包商城。
- 图片和附件使用社交专用上传接口，不复用文章上传权限入口。
- 图片大小限制为 5MB，附件大小限制为 50MB。
- 非互关用户只允许发送文本消息；图片和附件发送仅互关后开放。

后端设计：

- `social_message` 增加文件元数据字段：`file_id`、`file_url`、`file_name`、`file_size`、`file_content_type`。
- `content` 允许为空，用于图片和附件消息；文本消息仍要求内容非空且不超过 1000 字。
- `POST /api/social/upload/image`：上传社交图片，校验图片 MIME、后缀和 5MB 限制。
- `POST /api/social/upload/file`：上传社交附件，校验允许后缀、MIME 和 50MB 限制。
- `POST /api/social/messages` 按 `messageType` 分支校验：`TEXT` 校验文本和敏感词；`IMAGE`、`FILE` 校验文件归属、文件类型、互关状态和文件大小。

前端设计：

- 输入区提供 emoji、图片、附件、发送四类操作。
- emoji 面板插入常用 emoji 到当前输入内容末尾。
- 图片和附件上传成功后立即发送对应类型消息。
- 非互关时图片和附件按钮禁用，保留文本发送额度规则。
- 图片气泡展示缩略图并可打开链接预览；附件气泡展示文件名、大小和下载入口。
