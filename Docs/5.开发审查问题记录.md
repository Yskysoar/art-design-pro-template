## 一、文档目的

本文档记录当前项目在开发、审查和上线准备过程中发现的问题、风险、设计不足、修复状态、建议解决方案和验证方式。

### 2.1 状态说明

| 状态 | 含义 |
| :--- | :--- |

| 优先级 | 含义 |
| :--- | :--- |
| P0 | 上线阻断；不修复或未明确替代方案时不建议发布生产 |

## 三、系统基础与安全模块

### 3.1 默认开发账号和弱初始密码不能进入生产

| 项目 | 内容 |
| :--- | :--- |
| 优先级 | P0 |
| 问题内容 | 当前模板仍使用 `admin/admin123` 作为开发期管理员账号，登录页还内置演示账号快捷选择；新增用户未传密码时会使用默认初始密码。 |
| 证据范围 | `Code/art-design-pro/src/views/auth/login/index.vue`、后端 Mock SQL、`Docs/1.项目说明文档.md`。 |
| 产生原因 | 当前处于模板和本地联调阶段，需要稳定账号快速验证前后端链路。 |
| 业务影响 | 如果生产环境保留默认账号、默认密码或登录页演示账号，容易被扫描、撞库或误用。 |
| 当前状态 | 持续关注 |
| 解决方案 | 开发环境可保留；生产构建必须隐藏演示账号选择和自动填充逻辑。后端初始化脚本不得保留默认弱口令；管理员首次登录应强制改密，或由部署流程生成随机初始密码；新增用户应由管理员设置初始密码或生成一次性重置链接。 |
| 验证方式 | 上线前检查登录页生产构建、Mock SQL、部署脚本、配置文件和文档，确认不包含默认生产密码或演示账号提示。 |

### 3.2 Token 持久化在 localStorage 中，XSS 后存在泄露风险

| 项目 | 内容 |
| :--- | :--- |
| 优先级 | P1 |
| 问题内容 | 前端用户 Store 通过 `pinia-plugin-persistedstate` 使用 `localStorage` 持久化登录态和 token。 |
| 证据范围 | `Code/art-design-pro/src/store/modules/user.ts`、`Code/art-design-pro/src/utils/http/index.ts`。 |
| 产生原因 | 当前模板采用 Bearer JWT + 前端持久化方案，便于开发联调和刷新后恢复登录态。 |
| 业务影响 | 如果富文本、iframe、第三方脚本或其他 XSS 点被利用，攻击者可能读取 token 并冒用用户身份。 |
| 当前状态 | 持续关注 |
| 解决方案 | 上线前优先评估 HttpOnly + Secure + SameSite Cookie，并同步补 CSRF 防护；如果继续使用 Bearer JWT，应缩短 access token 有效期，refresh token 不暴露给普通脚本，补登出失效列表、设备管理和异常登录审计。 |
| 验证方式 | 做认证方案专项复核和 XSS 回归；确认 token 不再被普通脚本长期读取，或已记录并接受 Bearer 方案风险且有失效策略。 |

### 3.3 CORS 与 CSRF 生产策略需要显式收敛

| 项目 | 内容 |
| :--- | :--- |
| 优先级 | P1 |
| 问题内容 | 后端 `SecurityConfig` 当前禁用 CSRF，并使用默认 CORS 配置。 |
| 证据范围 | `Code/backend-template-java/src/main/java/com/template/security/config/SecurityConfig.java`。 |
| 产生原因 | 当前认证方式为 Bearer JWT，本地开发阶段优先保证前后端联调可用。 |
| 业务影响 | Bearer Token 场景可以不依赖 CSRF 防护，但生产环境如果 CORS 放开过宽，会扩大跨站调用和 token 滥用风险；后续如果切换 Cookie 认证，禁用 CSRF 会变成高风险。 |
| 当前状态 | 待修复 |
| 解决方案 | 增加生产环境 CORS 白名单配置，显式限制 Origin、Method、Header、Credentials 和暴露 Header；如果后续改为 Cookie 认证，则必须启用并配置 CSRF token。 |
| 验证方式 | 新增 CORS 配置测试或真实 HTTP 回归：白名单域名可访问，非白名单域名被拒绝；Cookie 方案补 CSRF 缺失和错误 token 用例。 |
