# 开发审查问题记录

最后更新：2026-06-01

## 一、文档目的

本文档用于记录开发过程中发现的业务审查问题，按业务模块归档，方便后续继续开发、测试和上线前回顾。

本文档只记录开发阶段发现的问题、原因、当前状态、修复方式和后续验证点；整体架构设计仍维护在 `3.后端开发设计方案.md`，测试设计仍维护在 `4.后端测试设计.md`。

## 二、状态说明

| 状态 | 含义 |
| :--- | :--- |
| 已修复 | 代码、数据或文档已完成整改，并已通过对应验证 |
| 部分修复 | 已完成主要整改，但仍有测试、文档或边界场景待补齐 |
| 待修复 | 问题已确认，尚未进入代码整改 |

| 持续关注 | 当前阶段可接受，但上线前需要再次复核 |

## 三、系统基础与安全模块

### 3.1 后端接口只做登录校验，缺少细粒度权限校验

| 项目 | 内容 |
| :--- | :--- |
| 问题内容 | 当前后端 Spring Security 主要限制为登录后可访问，配置项、用户、角色、菜单、组织等管理接口尚未按权限码或超级管理员身份做服务端细粒度校验。 |
| 产生原因 | 当前阶段优先完成认证、菜单、系统 CRUD 和前后端联调，权限码已经进入用户信息和菜单按钮模型，但接口级鉴权拦截尚未落地。 |
| 业务影响 | 普通后台用户如果拿到接口地址和 token，可能绕过前端按钮隐藏直接调用管理接口。前端权限只能改善体验，不能作为安全边界。 |
| 当前状态 | 部分修复 |
| 已完成内容 | 已新增 `PermissionService`，用户、角色、菜单管理、组织、配置项等系统管理接口已在 Controller 入口按权限码校验；超级管理员 `R_SUPER` 需要同时存在于 JWT 角色声明和数据库启用角色数据中，避免角色停用后旧 token 在过期前继续跳过权限码校验。Mock SQL 已补充 `system:user:view`、`system:role:view` 并授予超级管理员。 |
| 待修复内容 | 后续建议补 Controller/MockMvc 测试，覆盖无 token、无权限 token、有权限 token 三类响应；后续可视情况演进为注解式权限校验。 |
| 验证方式 | 已执行 `mvn test`，结果 `Tests run: 19, Failures: 0, Errors: 0, Skipped: 0`。 |

### 3.2 默认开发账号和弱初始密码不能进入生产

| 项目 | 内容 |
| :--- | :--- |
| 问题内容 | 当前模板使用 `admin/admin123` 作为开发期管理员账号，同时新增用户未传密码时会使用默认初始密码。 |
| 产生原因 | 当前项目处于模板和本地联调阶段，需要稳定账号快速验证前后端链路。 |
| 业务影响 | 如果生产环境沿用默认账号或默认密码，会形成明显安全风险。 |
| 当前状态 | 持续关注 |
| 修复方式 | 开发期保留，生产前改为首次启动强制修改管理员密码，或通过初始化脚本生成随机密码；新增用户应由管理员设置初始密码，或生成一次性重置链接。 |
| 验证方式 | 上线前检查 Mock SQL、配置文件、文档和部署脚本，确认不再包含默认生产密码。 |

## 四、配置项管理模块

### 4.1 配置项接口未限制为超级管理员可操作

| 项目 | 内容 |
| :--- | :--- |
| 问题内容 | 文档设计要求配置项只能由超级管理员维护，但当前 `/api/config/**` 只要登录即可调用。 |
| 产生原因 | 配置项 CRUD 先完成了业务主流程，接口级权限控制尚未接入。 |
| 业务影响 | 配置项会影响用户组织关系、角色层级、匿名访问等基础行为，错误开放会导致系统规则被非授权用户修改。 |
| 当前状态 | 已修复 |
| 修复方式 | 配置项列表、新增、更新、删除接口已统一校验 `system:config:manage`；超级管理员 `R_SUPER` 可操作，普通角色必须拥有该权限码。 |
| 验证方式 | 已执行后端单元测试和编译；后续重启后端后可追加真实接口 403 回归。 |

### 4.2 配置值缺少按配置键的业务校验

| 项目 | 内容 |
| :--- | :--- |
| 问题内容 | 当前配置值按普通字符串保存，没有按 `configKey` 限定枚举、布尔值、数字范围或 JSON 格式。 |
| 产生原因 | `sys_config` 当前采用通用 key-value 结构，先满足快速扩展，没有建立配置元数据和类型校验。 |
| 业务影响 | 例如 `user_org_relation_mode` 如果被写成非法值，后端不会按一对一规则限制用户组织关系，实际效果会退化为一对多。 |
| 当前状态 | 部分修复 |
| 已完成内容 | 后端已对内置配置键做保存前校验：`user_org_relation_mode=ONE_TO_ONE/ONE_TO_MANY`，`role_level_enabled=true/false`，`anonymous_portal_access=true/false`，`guest_admin_access=true/false`。 |
| 待修复内容 | 后续如配置项增多，建议抽象配置元数据表或配置定义枚举，返回配置类型给前端渲染专用控件。 |
| 验证方式 | 已新增配置项非法枚举单元测试，`mvn test` 通过。 |

### 4.3 配置键前端不可编辑，但后端仍允许修改

| 项目 | 内容 |
| :--- | :--- |
| 问题内容 | 前端编辑弹窗已经禁用配置键输入，但后端更新接口仍会根据请求体更新 `configKey`。 |
| 产生原因 | 前端交互和后端业务约束没有完全对齐，后端复用了新增和更新同一个请求对象。 |
| 业务影响 | 固定配置键被恶意或误操作改名后，依赖该键的业务逻辑会读取不到配置，导致规则失效或回退默认值。 |
| 当前状态 | 已修复 |
| 修复方式 | 更新配置项时后端会校验请求体 `configKey` 必须与数据库已有配置键一致，不一致返回“配置键不允许修改”。 |
| 验证方式 | 已新增单元测试覆盖修改配置键被拒绝，`mvn test` 通过。 |

### 4.4 逻辑删除与配置键唯一索引存在冲突

| 项目 | 内容 |
| :--- | :--- |
| 问题内容 | 配置项使用 MyBatis-Plus `deleteById` 做逻辑删除，但数据库对 `config_key` 建立单字段唯一索引。逻辑删除后记录仍占用唯一键，无法重新新增同名配置。 |
| 产生原因 | 业务层唯一校验排除了 `deleted=1`，但数据库唯一索引没有包含逻辑删除字段。 |
| 业务影响 | 用户在页面删除配置后，再次新增同名配置会通过业务唯一校验，却在数据库插入阶段失败，错误不够友好。 |
| 当前状态 | 部分修复 |
| 已完成内容 | 业务层唯一校验已改为不排除已删除记录，避免删除后新增同名配置通过业务校验却被数据库唯一索引拦截；系统内置配置已禁止删除。 |
| 待修复内容 | 普通扩展配置是否允许“删除后重建同名配置”仍需后续按业务确认；如果允许，需要调整数据库唯一索引策略。 |
| 验证方式 | 已新增系统内置配置禁止删除测试，`mvn test` 通过。 |

### 4.5 前端配置值控件过于通用

| 项目 | 内容 |
| :--- | :--- |
| 问题内容 | 配置值当前使用普通输入框，布尔值和枚举值也需要手动输入字符串。 |
| 产生原因 | 当前页面优先实现通用 CRUD，没有根据配置类型动态渲染表单控件。 |
| 业务影响 | 管理员容易输入大小写错误、空格或非法值，增加后端校验失败和误操作概率。 |
| 当前状态 | 待修复 |
| 修复方式 | 后端返回配置类型和允许值后，前端按类型渲染控件：布尔值使用开关，枚举值使用下拉框，数字使用数字输入框，长文本使用 textarea。 |
| 验证方式 | 前端类型检查通过；页面手动回归新增、编辑、搜索、删除；后端非法值仍能兜底拒绝。 |

## 五、用户与组织关联模块

### 5.1 用户组织关系模式依赖配置值，缺少非法配置兜底

| 项目 | 内容 |
| :--- | :--- |
| 问题内容 | 用户组织关系依赖 `user_org_relation_mode` 配置项；当前仅当配置值等于 `ONE_TO_ONE` 时限制一个用户一个组织，其他值会按一对多处理。 |
| 产生原因 | 配置项还没有类型校验，用户服务读取配置后没有对非法值主动报错。 |
| 业务影响 | 如果配置值被写错，一对一限制会失效。 |
| 当前状态 | 已修复 |
| 已完成内容 | 配置项保存阶段已限制 `user_org_relation_mode` 只能为 `ONE_TO_ONE` 或 `ONE_TO_MANY`；用户服务读取到非法值时会返回业务错误，不再静默退化为一对多。 |
| 验证方式 | 已新增配置项非法值测试和用户组织关系非法配置测试，`mvn test` 通过。 |

## 六、角色与权限模块

### 6.1 角色层级设计已进入文档，但后端强约束尚未落地

| 项目 | 内容 |
| :--- | :--- |
| 问题内容 | 文档要求角色层级后台可配置，并且只有超级管理员可以维护层级和基础权限；当前角色 CRUD、角色权限和角色数据权限已实现，但角色层级对管理动作的限制尚未完整实现。 |
| 产生原因 | 当前阶段优先实现角色列表、CRUD、菜单授权和数据权限，角色层级属于下一步权限闭环。 |
| 业务影响 | 如果不补齐层级约束，低层级管理员可能创建或修改高层级角色，破坏项目通用权限模型。 |

| 待修复内容 | 角色新增接口当前只创建固定低层级业务角色，后续如果开放后台配置角色层级，需要把层级字段纳入请求并增加更细测试。 |
| 验证方式 | 已执行 `mvn test` 通过；后续建议补角色服务独立单元测试。 |

## 九、Mock SQL 与数据库脚本

### 9.1 Mock SQL 需要随着模块开发持续维护

| :--- | :--- |
| 问题内容 | Mock SQL 已包含用户、角色、菜单、权限、组织、配置项和关联关系，但每个模块新增字段或新增业务规则后，都需要同步维护初始化数据。 |

## 十二、维护规则

- 每次完成一个业务模块后，应把审查中发现的问题补充到本文档。
- 如果问题已修复，应补充修复方式、测试方式和结果。
- 如果问题影响接口契约，应同步更新 `1.项目说明文档.md` 和 `3.后端开发设计方案.md`。
- 如果问题影响测试策略，应同步更新 `4.后端测试设计.md`。
- 如果问题属于上线安全风险，应同步更新 `2.代码审查总结.md`。
