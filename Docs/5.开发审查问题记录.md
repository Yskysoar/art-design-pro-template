# 开发审查问题记录

最后更新：2026-06-02

## 一、文档目的

本文档用于记录开发过程中发现的业务审查问题，按业务模块归档，方便后续继续开发、测试和上线前回顾。

本文档只记录开发阶段发现的问题、原因、当前状态、修复方式和后续验证点；整体架构设计仍维护在 `3.后端开发设计方案.md`，测试设计仍维护在 `4.后端测试设计.md`。

## 二、状态说明

| 状态 | 含义 |
| :--- | :--- |
| 待修复 | 问题已确认，尚未进入代码整改 |
| 部分修复 | 已完成主要整改，但仍有测试、文档或边界场景待补齐 |
| 持续关注 | 当前阶段可接受，但上线前需要再次复核 |
| 设计待确认 | 需要用户确认业务规则后再实现 |
| 已修复 | 代码、数据或文档已完成整改，并已通过对应验证 |

模块内排序规则：待修复、部分修复、持续关注、设计待确认的问题放在前面；已修复问题放在该模块结尾。

## 三、系统基础与安全模块

### 3.1 默认开发账号和弱初始密码不能进入生产

| 项目 | 内容 |
| :--- | :--- |
| 问题内容 | 当前模板使用 `admin/admin123` 作为开发期管理员账号，同时新增用户未传密码时会使用默认初始密码。 |
| 产生原因 | 当前项目处于模板和本地联调阶段，需要稳定账号快速验证前后端链路。 |
| 业务影响 | 如果生产环境沿用默认账号或默认密码，会形成明显安全风险。 |
| 当前状态 | 持续关注 |
| 修复方式 | 开发期保留；生产前改为首次启动强制修改管理员密码，或通过初始化脚本生成随机密码；新增用户应由管理员设置初始密码，或生成一次性重置链接。 |
| 验证方式 | 上线前检查 Mock SQL、配置文件、文档和部署脚本，确认不再包含默认生产密码。 |

### 3.2 Token 持久化在 localStorage 中，XSS 后存在泄露风险

| 项目 | 内容 |
| :--- | :--- |
| 问题内容 | 前端用户 Store 通过 `pinia-plugin-persistedstate` 使用 `localStorage` 持久化登录态和 token。 |
| 产生原因 | 当前模板采用 Bearer JWT + 前端持久化方案，便于开发联调和刷新后恢复登录态。 |
| 业务影响 | 如果富文本、iframe、第三方脚本或其他 XSS 点被利用，攻击者可能读取 localStorage 中的 token 并冒用用户身份。 |
| 当前状态 | 持续关注 |
| 修复方式 | 开发阶段可继续保留；上线前评估改为 HttpOnly Cookie + SameSite + CSRF 防护，或缩短 token 有效期并配合 refresh token、设备管理、登出失效列表。 |
| 验证方式 | 上线前做 XSS 回归和认证方案复核；确认 token 不再被普通脚本长期读取，或已接受 Bearer 方案的风险并补齐失效策略。 |

### 3.3 CORS 与 CSRF 生产策略需要显式收敛

| 项目 | 内容 |
| :--- | :--- |
| 问题内容 | 后端当前在 `SecurityConfig` 中禁用 CSRF，并使用默认 CORS 配置。 |
| 产生原因 | 当前认证方式为 Bearer JWT，本地开发阶段优先保证前后端联调可用。 |
| 业务影响 | Bearer Token 场景可以不依赖 CSRF 防护，但生产环境如果 CORS 放开过宽，会扩大跨站调用和 token 滥用风险。 |
| 当前状态 | 待修复 |
| 修复方式 | 生产配置中显式限制允许的 Origin、Method、Header 和暴露 Header；如果后续改为 Cookie 认证，则必须重新启用并配置 CSRF 防护。 |
| 验证方式 | 新增 CORS 配置测试或真实 HTTP 回归：允许域名可访问，非白名单域名被拒绝；Cookie 方案需补 CSRF 失败用例。 |

### 3.4 后端接口只做登录校验，缺少细粒度权限校验

| 项目 | 内容 |
| :--- | :--- |
| 问题内容 | 早期后端 Spring Security 主要限制为登录后可访问，配置项、用户、角色、菜单、组织等管理接口尚未按权限码或超级管理员身份做服务端细粒度校验。 |
| 产生原因 | 早期阶段优先完成认证、菜单、系统 CRUD 和前后端联调，权限码先进入用户信息和菜单按钮模型，接口级鉴权后续补齐。 |
| 业务影响 | 普通后台用户如果拿到接口地址和 token，可能绕过前端按钮隐藏直接调用管理接口。前端权限只能改善体验，不能作为安全边界。 |
| 当前状态 | 已修复 |
| 修复方式 | 已新增 `PermissionService`，用户、角色、菜单管理、组织、配置项等系统管理接口已在 Controller 入口按权限码校验；超级管理员 `R_SUPER` 需要同时存在于 JWT 声明和数据库启用角色数据中，避免角色停用后旧 token 在过期前继续跳过权限码校验；权限不足的业务异常已映射为 HTTP 403。Mock SQL 已补充 `system:user:view`、`system:role:view` 并授予超级管理员。 |
| 验证方式 | 已执行 `mvn test`，系统管理 Controller 权限入口测试覆盖 401、403、200 和主要写接口；真实角色账号联调已验证 `admin` 系统管理接口 200，`moderator` 系统管理接口 403。 |

## 四、配置项管理模块

### 4.1 前端配置值控件过于通用

| 项目 | 内容 |
| :--- | :--- |
| 问题内容 | 配置值当前使用普通输入框，布尔值和枚举值也需要手动输入字符串。 |
| 产生原因 | 当前页面优先实现通用 CRUD，没有根据配置类型动态渲染表单控件。 |
| 业务影响 | 管理员容易输入大小写错误、空格或非法值，增加后端校验失败和误操作概率。 |
| 当前状态 | 待修复 |
| 修复方式 | 后端返回配置类型和允许值后，前端按类型渲染控件：布尔值使用开关，枚举值使用下拉框，数字使用数字输入框，长文本使用 textarea。 |
| 验证方式 | 前端类型检查通过；页面手动回归新增、编辑、搜索、删除；后端非法值仍能兜底拒绝。 |

### 4.2 配置值缺少按配置键的业务校验

| 项目 | 内容 |
| :--- | :--- |
| 问题内容 | 配置值按普通字符串保存，没有按 `configKey` 限定枚举、布尔值、数字范围或 JSON 格式。 |
| 产生原因 | `sys_config` 采用通用 key-value 结构，先满足快速扩展，没有建立完整配置元数据。 |
| 业务影响 | 例如 `user_org_relation_mode` 如果被写成非法值，后端不会按一对一规则限制用户组织关系，实际效果会退化为一对多。 |
| 当前状态 | 部分修复 |
| 已完成内容 | 后端已对内置配置键做保存前校验：`user_org_relation_mode=ONE_TO_ONE/ONE_TO_MANY`，`role_level_enabled=true/false`，`anonymous_portal_access=true/false`，`guest_admin_access=true/false`。角色层级设计已修正为“模板支持多级角色设计，由超级管理员按业务场景配置和维护”，不固定某一套层级模型。 |
| 待修复内容 | 后续如配置项增多，建议抽象配置元数据表或配置定义枚举，返回配置类型给前端渲染专用控件。 |
| 验证方式 | 已新增配置项非法枚举单元测试，`mvn test` 通过。 |

### 4.3 配置项逻辑删除的并发错误语义仍需优化

| 项目 | 内容 |
| :--- | :--- |
| 问题内容 | 配置项已允许逻辑删除后重新创建同名配置键，但并发重复新增时主要依赖数据库唯一索引兜底。 |
| 产生原因 | 当前优先修复业务正确性，数据库异常到友好业务错误的统一转换尚未专项处理。 |
| 业务影响 | 并发提交相同 `configKey` 时，用户可能看到数据库层异常或不够友好的错误提示。 |
| 当前状态 | 部分修复 |
| 已完成内容 | 数据库唯一索引已调整为 `(config_key, deleted)`；配置项实体使用 `@TableLogic(value = "0", delval = "id")`，删除后将 `deleted` 写为当前记录 ID；业务唯一校验只检查 `deleted=0` 的活跃配置。 |
| 待修复内容 | 把 `DuplicateKeyException` 转换为更友好的业务错误，并在更新、删除时检查影响行数。后续如引入 Flyway/Liquibase，需要补已有库迁移脚本。 |
| 验证方式 | 已补 Service 单元测试；`mvn test` 通过；真实接口已验证新增、删除、同名重建成功，重复活跃键仍返回业务错误；测试后已重置 `backend_template` 数据库。 |

### 4.4 配置项接口未限制为超级管理员可操作

| 项目 | 内容 |
| :--- | :--- |
| 问题内容 | 文档设计要求配置项只能由超级管理员维护，但早期 `/api/config/**` 只要登录即可调用。 |
| 产生原因 | 配置项 CRUD 先完成了业务主流程，接口级权限控制后续接入。 |
| 业务影响 | 配置项会影响用户组织关系、前台匿名访问、后台入口等基础行为，错误开放会导致系统规则被非授权用户修改。 |
| 当前状态 | 已修复 |
| 修复方式 | 配置项列表、新增、更新、删除接口已统一校验 `system:config:manage`；超级管理员 `R_SUPER` 可操作，普通角色必须拥有该权限码。 |
| 验证方式 | 已执行后端单元测试和编译；真实角色账号联调已验证无权限角色访问系统管理接口返回 403。 |

### 4.5 配置键前端不可编辑，但后端仍允许修改

| 项目 | 内容 |
| :--- | :--- |
| 问题内容 | 前端编辑弹窗已经禁用配置键输入，但早期后端更新接口仍会根据请求体更新 `configKey`。 |
| 产生原因 | 前端交互和后端业务约束没有完全对齐，后端复用了新增和更新同一个请求对象。 |
| 业务影响 | 固定配置键被恶意或误操作改名后，依赖该键的业务逻辑会读取不到配置，导致规则失效或回退默认值。 |
| 当前状态 | 已修复 |
| 修复方式 | 更新配置项时后端会校验请求体 `configKey` 必须与数据库已有配置键一致，不一致返回“配置键不允许修改”。 |
| 验证方式 | 已新增单元测试覆盖修改配置键被拒绝，`mvn test` 通过。 |

### 7.3 动态路由初始化期间快速切换页面会造成跳转慢和回弹

| 项目 | 内容 |
| :--- | :--- |
| 问题内容 | 早期前端在动态路由未注册时会串行请求用户信息和菜单，并在初始化进行中对后续导航执行 `next(false)`。用户快速切换页面时，页面可能被取消、回弹或表现为响应慢；如果期间任意接口返回 401，前端会触发全局退出登录，用户体感像“来回跳转导致 token 失效”。 |
| 产生原因 | 路由初始化任务未被复用，初始化中的新导航没有等待同一任务完成；401 拦截器对并发未授权错误缺少单次退出保护。 |
| 业务影响 | 首次进入后台、刷新后恢复页面或快速点击菜单时，可能出现全屏 loading 时间偏长、导航取消、旧导航反抢新导航、重复重置动态路由和重复跳登录页。 |
