## 一、项目概况

当前项目是一个 Vue 3 后台管理系统前端模板，定位为后续业务系统的前端基础工程。项目已经包含登录、权限、动态路由、布局、表格、表单、图表、上传、富文本、文章管理、系统管理等后台系统常见能力。

后续开发计划是在当前前端项目基础上补充自研后端，因此需要保持接口契约清晰、Mock 数据可控、权限模型稳定，并在上线前逐步替换演示逻辑。

## 二、业务功能

### 2.1 认证与账号

当前包含登录、注册、找回密码页面。登录页调用 `POST /api/auth/login`，成功后写入用户状态并跳转目标页面。注册页已接入真实接口 `POST /api/auth/register`，注册成功后创建普通用户角色 `R_USER` 并绑定默认组织 `ORG_TEMPLATE`。忘记密码页已接入开发期真实重置接口 `POST /api/auth/reset-password`，通过账号和新密码完成重置；生产上线前必须补验证码、邮箱或短信校验、频率限制和审计日志。

核心状态位于 `src/store/modules/user.ts`：

- `isLogin`：登录状态。
- `accessToken` / `refreshToken`：访问令牌和刷新令牌。
- `info`：当前用户资料。
- `roles` / `buttons`：通过用户信息参与路由和按钮权限控制。

右上角用户菜单展示当前登录用户真实头像，未配置头像时使用本地默认头像兜底。菜单保留个人中心、修改密码、锁屏和退出登录；使用文档、GitHub 入口不再放在右上角用户菜单。修改密码通过弹窗完成，成功后清理登录状态并要求用户使用新密码重新登录。

所有和当前登录用户对接的前端展示点必须使用 `/api/user/info` 写入 `userStore.getUserInfo` 后的真实数据渲染，包括头像、用户名、邮箱、角色、个人中心、锁屏弹窗和右上角用户菜单。只有后端未返回字段时，才允许使用本地默认头像或空状态文案兜底。

### 2.2 权限与菜单

项目支持两种权限模式：

| 模式 | 环境变量 | 说明 |
| :--- | :--- | :--- |
| 前端权限 | `VITE_ACCESS_MODE=frontend` | 前端维护完整路由，根据 `user.info.roles` 过滤可访问菜单。 |
| 后端权限 | `VITE_ACCESS_MODE=backend` | 后端返回当前用户可访问菜单树，前端按菜单树注册动态路由。 |

当前项目默认使用后端权限模式，`.env.development` 中应保持 `VITE_ACCESS_MODE=backend`。正式业务建议继续采用后端权限模式：后端根据用户、角色、菜单、按钮权限关系返回可访问菜单和按钮标识，前端负责展示和路由注册，后端接口仍必须做权限校验。当前系统管理页面按钮已统一使用后端 `system:*` 权限码，例如 `system:user:add`、`system:role:permission`、`system:config:manage`。

模板支持多级角色与权限结构。具体层级规则由超级管理员根据业务场景配置和维护，例如场馆系统中一个场馆只能有一个拥有者，拥有者可以设置多个场馆管理员，普通用户只负责预定。角色层级用于表达“谁能管理谁”和业务管理边界；后端仍需用权限码和数据权限做强校验。

前台和后台共用同一套认证后端。前端登录后根据当前用户角色和访问端范围决定进入页面：普通用户只能进入前台；商家、版主等双端角色可进入前台和对应后台，并可在右上角的角色或工作台入口中切换当前访问视图。

### 2.3 系统管理

当前侧边栏系统管理包含用户管理、角色管理、组织管理、配置管理、菜单管理和敏感词库页面。顶部栏右上角齿轮保留原模板设置抽屉能力，不进入侧边栏菜单。后端需要支持：

- 用户分页查询、创建、更新、删除、启停用。
- 角色分页查询、创建、更新、删除、启停用。
- 角色授权菜单和按钮权限。
- 菜单树维护。
- 组织树查询、创建、更新、删除，并由后端保护存在子组织、用户关联、角色数据权限关联的组织；组织树查询已按当前用户角色数据权限过滤。
- 配置分页查询、创建、更新、删除，并限制不可编辑配置的修改和删除；系统内置配置不允许删除，内置配置值需要符合后端枚举或布尔约束；普通扩展配置逻辑删除后允许重新创建同名 `configKey`。
- 敏感词库分页查询、创建、更新、启停用和删除；敏感词命中后用于拦截文章标题、摘要、正文、评论、用户信息、组织、角色、配置和菜单等文本入口。

用户管理页展示规则：

- 用户状态筛选和表格状态统一使用“启用/禁用”语义；`PATCH /api/user/{id}/status` 用于临时启用或禁用账号，数据仍保留。
- `DELETE /api/user/{id}` 表示逻辑删除用户，不再作为临时“禁用/注销”入口。页面操作文案统一为“删除用户”，并提示如需临时停用请使用禁用按钮。
- 手机号列应保持单行展示，避免窄表格中号码换行影响阅读。
- 用户列表接口返回 `orgIds`，前端通过 `/api/org/tree` 映射为组织名称展示；找不到的组织 ID 使用 `#id` 兜底，便于发现脏数据。

敏感词库页展示规则：

- 状态列使用和其他系统管理页一致的标签显示“启用/禁用”，不使用表格内开关。
- 启用或禁用敏感词通过操作列按钮触发。
- 页面复用系统管理统一搜索表单、表格头部、表格操作按钮和弹窗表单样式。

### 2.4 文章管理

当前包含文章列表、文章发布、文章详情、评论等页面。正式业务建议拆分为：

- 文章分类接口。
- 文章分页列表。
- 文章详情。
- 新增、编辑、删除文章。
- 发布、下架文章。
- 文章附件关联，一篇文章可夹带多个上传附件。
- 文章详情底部评论区，参考 CSDN 类文章详情模式展示用户评论。
- 评论分页、回复、隐藏或删除。

当前开发阶段已实现文章分类、列表、详情、新增、编辑、删除、发布/草稿/下架、附件关联和文章详情底部评论区。评论模块定位不是独立留言墙，而是文章详情底部的公开评论区。只要用户能打开当前文章，就可以查看该文章评论；评论内容使用纯文本展示，不直接渲染富 HTML。

项目内置常见后台组件能力：

## 三、核心业务逻辑

### 3.1 登录流程

```text
用户输入账号密码
-> 表单校验
-> POST /api/auth/login
-> 后端返回 token / refreshToken
-> userStore.setToken()
-> userStore.setLoginStatus(true)
-> 跳转 redirect 或首页
-> 路由守卫初始化用户信息和动态路由
```

### 3.2 路由初始化流程

```text
router.beforeEach
-> 判断是否已登录或是否访问静态路由
-> 未登录则跳转登录页
-> 已登录且动态路由未注册
-> 初始化动态路由
   - backend：并发请求 GET /api/user/info 和 GET /api/v3/system/menus
   - frontend：先请求 GET /api/user/info，再根据 user.info.roles 过滤本地 asyncRoutes
-> 注册动态路由
-> 保存菜单和 iframe 路由
-> 校验目标路由权限
-> 重新导航到目标页面
```

动态路由初始化期间复用同一个初始化任务，避免快速点击多个页面时重复请求用户信息和菜单。初始化完成后只允许最新一次导航恢复跳转，旧导航取消，防止旧页面反抢最新页面。`frontend` 权限模式依赖用户角色过滤本地路由，因此不能和用户信息请求并发。

动态菜单数据在菜单树和权限校验中统一使用完整路径，例如 `/system/user`；注册到 Vue Router 时，子路由会转换为父级下的相对路径，例如 `user`，避免绝对子路由脱离父级导致登录后跳转 500 或动态路由加载失效。登录页只接受业务页面作为 `redirect`，登录页、错误页或无效路径会回退到 `/`，再由守卫跳到当前用户首页。

### 3.3 请求处理流程

`src/utils/http/index.ts` 对 Axios 做统一封装：

- 请求拦截器附加 `Authorization`，并设置 JSON 请求头。
- 响应拦截器要求后端返回 `{ code, msg, data }`。
- `code=200` 返回 `data`。
- `code=401` 自动退出登录；同一轮未授权错误只触发一次退出和一次路由重置，避免多个并发请求反复清理登录态。
- `code=403` 或 HTTP 403 转换为 `HttpError` 并提示无权限。
- 其他业务错误转换为 `HttpError` 并提示。

## 四、后端对接规范

### 4.1 通用响应格式

```ts
interface BaseResponse<T = unknown> {
  code: number
  msg: string
  data: T
}
```

| code | 含义 | 前端行为 |
| :--- | :--- | :--- |
| 200 | 成功 | 返回 `data` |
| 400 | 参数错误 | 显示 `msg` |
| 401 | 未登录或 token 失效 | 自动退出并跳转登录页 |
| 403 | 无权限 | 显示 `msg` 或跳转 403 |
| 404 | 资源不存在 | 显示 `msg` |
| 500 | 服务端错误 | 显示 `msg` |

### 4.2 认证接口

#### POST `/api/auth/login`

请求体：

```ts
interface LoginParams {
  userName: string
  password: string
}
```

响应：

```ts
interface LoginResponse {
  token: string
  refreshToken?: string
}
```

建议：

- 登录参数放在请求 body，不放 URL query。
- 生产环境必须使用 HTTPS。
- 如后端采用 Bearer 认证，前端请求头统一为 `Authorization: Bearer <token>`。
- 如后端采用 HttpOnly Cookie，前端应开启 `withCredentials`，并移除本地 token 持久化。

#### POST `/api/auth/register`

请求体：

```ts
interface RegisterParams {
  userName: string
  password: string
}
```

规则：

- 用户名 3-20 位，只允许英文字母、数字和下划线。
- 密码至少 6 位。
- 注册用户默认分配 `R_USER` 普通用户角色，并绑定 `ORG_TEMPLATE` 默认组织。
- 重复用户名返回业务错误。

#### POST `/api/auth/reset-password`

请求体：

```ts
interface ResetPasswordParams {
  userName: string
  newPassword: string
}
```

规则：

- 当前为开发期找回密码方案，通过账号和新密码重置。
- 新密码不能与当前密码相同。
- 生产上线前必须增加验证码、邮箱或短信校验、频率限制和审计日志。

#### GET `/api/user/info`

响应：

```ts
interface UserInfo {
  userId: number | string
  userName: string
  email?: string
  avatar?: string
  roles: string[]
  buttons: string[]
}
```

### 4.3 菜单接口

后端权限模式需要实现：

#### GET `/api/v3/system/menus`

```ts
interface AppRouteRecord {
  id?: number | string
  path: string
  name?: string
  redirect?: string
  component?: string
  meta: {
    title: string
    icon?: string
    roles?: string[]
    authList?: Array<{ authMark: string; title?: string }>
    isHide?: boolean
    isHideTab?: boolean
    keepAlive?: boolean
    isIframe?: boolean
    link?: string
    activePath?: string
    fixedTab?: boolean
    isFullPage?: boolean
  }
  children?: AppRouteRecord[]
}
```

注意：

- `component` 应为前端 `src/views` 下的组件路径，例如 `/system/user`。
- iframe 菜单必须设置 `meta.isIframe=true` 和 `meta.link`。
- 上线前应对 iframe link 做域名白名单控制。
- 菜单树只返回当前用户有权限访问的节点。

### 4.4 用户和角色接口建议

#### 用户组织关联

用户基础资料与组织关系建议通过独立关联表 `sys_user_org` 维护，后端返回用户列表时应同时返回当前用户关联的组织 ID 集合，便于前端在编辑弹窗中回填组织树。

建议接口：

| 接口 | 方法 | 说明 |
| :--- | :--- | :--- |
| `/api/user/list` | GET | 用户分页列表，返回角色编码和组织 ID 集合 |
| `/api/user` | POST | 新增用户时同时保存用户-组织关系 |
| `/api/user/{id}` | PUT | 更新用户基础资料、角色和组织关系 |
| `/api/user/{id}/orgs` | GET | 查询某个用户当前绑定的组织集合 |
| `/api/user/{id}/orgs` | PUT | 保存某个用户的组织集合 |
| `/api/user/profile/password` | PUT | 当前登录用户修改自己的密码，需校验当前密码 |

前端编辑用户时应支持：

- 角色多选。
- 组织树多选。
- 当 `user_org_relation_mode=ONE_TO_ONE` 时，只允许选择一个组织。
- 当 `user_org_relation_mode=ONE_TO_MANY` 时，允许选择多个组织。

若后端返回的组织模式为一对一，前端应将多选树切换为单选展示，但接口保持统一的 `orgIds` 数组格式。

```text
GET    /api/user/list              用户分页
POST   /api/user                   新增用户
PUT    /api/user/{id}              更新用户
DELETE /api/user/{id}              删除用户
PATCH  /api/user/{id}/status       启用/停用用户
PUT    /api/user/profile/password  当前登录用户修改自己的密码

GET    /api/role/list              角色分页
POST   /api/role                   新增角色
PUT    /api/role/{id}              更新角色
DELETE /api/role/{id}              删除角色
GET    /api/role/{id}/permissions  查询角色权限
PUT    /api/role/{id}/permissions  保存角色权限
```

分页响应建议：

```ts
interface PageResult<T> {
  records: T[]
  current: number
  size: number
  total: number
}
```

### 4.5 文件上传接口建议

```text
POST /api/common/upload
POST /api/common/upload/image
POST /api/common/upload/file
POST /api/common/upload/wangeditor
GET  /api/common/files/**
```

```ts
interface UploadResponse {
  url: string
  fileName?: string
  size?: number
  contentType?: string

```

安全要求：

- 后端校验文件 MIME、后缀、大小。
- 图片重编码或本地存储安全校验。
- 上传目录不允许直接执行脚本。
- 富文本上传返回的 URL 必须可信。
- 当前先使用后端本地 `uploads/` 存储，返回 `/api/common/files/**` 受控访问地址，不暴露服务器绝对路径。
- 上传接口需要登录并校验上传权限，读取文件接口只允许访问已落库且未逻辑删除的资源。

文章评论列表响应建议：

```ts
interface ArticleCommentItem {
  id: number
  articleId: number
  parentId: number
  rootId: number
  content: string
  status: 'NORMAL' | 'HIDDEN' | 'DELETED'
  userName: string
  userAvatar?: string
  replyToUserName?: string
  createTime: string
  mine?: boolean
  canHide?: boolean
  canRestore?: boolean
  canDelete?: boolean
  replies?: ArticleCommentItem[]

```

评论规则：

- `GET /api/article/comment/list` 为公开只读接口；只要当前文章可被访问，就可查看评论区内容。
- `POST /api/article/comment` 用于新增评论或回复；仅登录用户可发布，游客只能查看公开评论。
- `DELETE /api/article/comment/{id}` 管理端删除评论时不物理删除，状态改为 `DELETED`，内容隐藏，子回复保留。
- `PATCH /api/article/comment/{id}/status` 用于管理端隐藏、恢复或删除评论。
- 评论内容按纯文本处理，前端使用文本插值展示，后端保存前做空值、500 字长度上限和控制字符校验。
- 文章列表和文章详情返回 `commentCount`，由 `article.comment_count` 冗余字段维护；一级评论和所有回复都计入评论总数，隐藏评论仍计入数量，删除评论不计入数量。文章列表只展示评论总数，不拆分主评论和回复；文章详情评论区标题展示 `X 条主评论 · Y 条回复`。
- 文章详情读取成功后递增文章浏览数；文章列表从详情页返回或重新激活时刷新列表数据，确保浏览数跟随最新后端值展示。
- 删除评论只隐藏当前节点内容，挂载在已删除评论下的历史子回复继续保留记录并可在评论树中展示，避免评论上下文断裂。
- 评论允许回复任意一条未删除评论；前端仍保持两层展示，所有子回复挂在一级评论下，回复子回复时显示“回复某人”，避免无限缩进。
- 一级评论下回复数量超过 3 条时，默认折叠为 3 条，并提供“展开全部 X 条回复”和“收起”切换。
- 评论隐藏功能由系统配置 `article_comment_hide_enabled` 控制，默认开启；只有超级管理员可修改该配置。超级管理员始终可隐藏、恢复和删除；普通管理员仅在配置开启时可隐藏、恢复和删除；文章发布者仅在配置开启时可隐藏或恢复自己文章下的评论，不能删除；评论作者只能删除自己的评论，不能隐藏或恢复。

权限约定：

| 权限码 | 用途 |
| :--- | :--- |
| `article:publish:add` | 新增文章、发布文章 |
| `article:publish:edit` | 编辑文章、删除文章、下架文章 |
| `article:upload` | 上传封面、富文本图片和附件 |
| `article:comment:manage` | 管理端隐藏、恢复或删除评论 |

## 五、新增接口开发方式

### 5.1 新增类型

在 `src/types/api/api.d.ts` 中补充接口参数和响应类型：

```ts
declare namespace Api {
  namespace User {
    interface QueryParams {
      userName?: string
      current?: number
      size?: number
    }

    interface UserItem {
      id: number
      userName: string
      status: string
    }
  }
}
```

### 5.2 新增请求函数

在 `src/api/` 下按业务模块创建或更新文件：

```ts
import request from '@/utils/http'

export function fetchUserList(params: Api.User.QueryParams) {
  return request.get<PageResult<Api.User.UserItem>>({
    url: '/api/user/list',
    params
  })
}
```

### 5.3 页面中使用

表格类页面优先使用现有 `useTable`：

```ts
const { tableData, loading, pagination, search } = useTable({
  apiFn: fetchUserList,
  defaultParams: {
    current: 1,
    size: 10
  }
})
```

## 六、本地 Mock 方案

当前已接入 Vite middleware 本地 Mock：

```text
mock/local-mock.ts
```

开发环境通过 `.env.development` 控制：

```env
VITE_USE_LOCAL_MOCK = true
```

该值为 `true` 时，Vite 开发服务器会直接响应已覆盖的 `/api/*` 请求，不再转发到外部 Mock 服务。生产构建不会启用该 middleware。

当前已覆盖接口：

| 接口 | 方法 | 说明 |
| :--- | :--- | :--- |
| `/api/auth/login` | POST | 返回本地登录 token |

| `/api/article/types` | GET | 返回本地文章分类列表 |
| `/api/article/detail` | GET | 返回本地文章详情，用于替代外部 JSON |

文章与上传真实后端完成后，当前前端已切换到 `src/api/article.ts` 和 `src/api/upload.ts`，本地 Mock 只保留开发兜底，不再作为主路径。

Mock 维护规则：

- Mock 数据必须与后端接口契约一致。
- Mock 只用于开发和调试，不进入生产逻辑。
- 不在 Mock 中保存真实密码、真实 token、真实用户隐私。
- 菜单、角色、按钮权限 Mock 至少覆盖管理员和普通用户两类场景。
