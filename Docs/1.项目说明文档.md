## 一、项目概况

当前项目是一个 Vue 3 后台管理系统前端模板，定位为后续业务系统的前端基础工程。项目已经包含登录、权限、动态路由、布局、表格、表单、图表、上传、富文本、文章管理、系统管理等后台系统常见能力。

后续开发计划是在当前前端项目基础上补充自研后端，因此需要保持接口契约清晰、Mock 数据可控、权限模型稳定，并在上线前逐步替换演示逻辑。

## 二、业务功能

### 2.1 认证与账号

当前包含登录、注册、找回密码页面。登录页调用 `POST /api/auth/login`，成功后写入用户状态并跳转目标页面。注册和找回密码当前偏演示性质，后续应由后端提供真实接口。

核心状态位于 `src/store/modules/user.ts`：

- `isLogin`：登录状态。
- `accessToken` / `refreshToken`：访问令牌和刷新令牌。
- `info`：当前用户资料。
- `roles` / `buttons`：通过用户信息参与路由和按钮权限控制。

### 2.2 权限与菜单

项目支持两种权限模式：

| 模式 | 环境变量 | 说明 |
| :--- | :--- | :--- |
| 前端权限 | `VITE_ACCESS_MODE=frontend` | 前端维护完整路由，根据 `user.info.roles` 过滤可访问菜单。 |
| 后端权限 | `VITE_ACCESS_MODE=backend` | 后端返回当前用户可访问菜单树，前端按菜单树注册动态路由。 |

正式业务建议采用后端权限模式：后端根据用户、角色、菜单、按钮权限关系返回可访问菜单和按钮标识，前端负责展示和路由注册，后端接口仍必须做权限校验。

### 2.3 系统管理

当前包含用户管理、角色管理、菜单管理页面。后端需要支持：

- 用户分页查询、创建、更新、删除、启停用。
- 角色分页查询、创建、更新、删除、启停用。
- 角色授权菜单和按钮权限。
- 菜单树维护。

### 2.4 文章管理

当前包含文章列表、文章发布、文章详情、评论等页面。正式业务建议拆分为：

- 文章分类接口。
- 文章分页列表。
- 文章详情。
- 新增、编辑、删除文章。
- 发布、下架文章。
- 评论分页、回复、删除。

### 2.5 组件能力

项目内置常见后台组件能力：

- 表格：分页、查询、排序、列配置、缓存。
- 表单：动态配置、响应式布局、校验。
- 上传：图片上传、富文本图片上传、Excel 导入导出。
- 图表：柱状图、折线图、环形图、地图、K 线等。
- 布局：工作标签页、面包屑、全局搜索、设置面板、锁屏。

后续业务开发应优先复用现有组件，避免重复实现表格、表单、权限、请求封装。

## 三、核心业务逻辑

### 3.1 登录流程

```text
用户输入账号密码
-> 表单校验
-> POST /api/auth/login
-> 后端返回 token / refreshToken
-> userStore.setToken()
-> userStore.setLoginStatus(true)
-> 跳转 redirect 或首页
-> 路由守卫初始化用户信息和动态路由
```

### 3.2 路由初始化流程

```text
router.beforeEach
-> 判断是否已登录或是否访问静态路由
-> 未登录则跳转登录页
-> 已登录且动态路由未注册
-> GET /api/user/info 获取用户信息
-> 根据权限模式获取菜单
   - frontend：过滤本地 asyncRoutes
   - backend：请求 /api/v3/system/menus
-> 注册动态路由
-> 保存菜单和 iframe 路由
-> 校验目标路由权限
-> 重新导航到目标页面
```

### 3.3 请求处理流程

`src/utils/http/index.ts` 对 Axios 做统一封装：

- 请求拦截器附加 `Authorization`，并设置 JSON 请求头。
- 响应拦截器要求后端返回 `{ code, msg, data }`。
- `code=200` 返回 `data`。
- `code=401` 自动退出登录。
- 其他业务错误转换为 `HttpError` 并提示。

## 四、后端对接规范

### 4.1 通用响应格式

```ts
interface BaseResponse<T = unknown> {
  code: number
  msg: string
  data: T
}
```

| code | 含义 | 前端行为 |
| :--- | :--- | :--- |
| 200 | 成功 | 返回 `data` |
| 400 | 参数错误 | 显示 `msg` |
| 401 | 未登录或 token 失效 | 自动退出并跳转登录页 |
| 403 | 无权限 | 显示 `msg` 或跳转 403 |
| 404 | 资源不存在 | 显示 `msg` |
| 500 | 服务端错误 | 显示 `msg` |

### 4.2 认证接口

#### POST `/api/auth/login`

请求体：

```ts
interface LoginParams {
  userName: string
  password: string
}
```

响应：

```ts
interface LoginResponse {
  token: string
  refreshToken?: string
}
```

建议：

- 登录参数放在请求 body，不放 URL query。
- 生产环境必须使用 HTTPS。
- 如后端采用 Bearer 认证，前端请求头统一为 `Authorization: Bearer <token>`。
- 如后端采用 HttpOnly Cookie，前端应开启 `withCredentials`，并移除本地 token 持久化。

#### GET `/api/user/info`

响应：

```ts
interface UserInfo {
  userId: number | string
  userName: string
  email?: string
  avatar?: string
  roles: string[]
  buttons: string[]
}
```

### 4.3 菜单接口

后端权限模式需要实现：

#### GET `/api/v3/system/menus`

```ts
interface AppRouteRecord {
  id?: number | string
  path: string
  name?: string
  redirect?: string
  component?: string
  meta: {
    title: string
    icon?: string
    roles?: string[]
    authList?: Array<{ authMark: string; title?: string }>
    isHide?: boolean
    isHideTab?: boolean
    keepAlive?: boolean
    isIframe?: boolean
    link?: string
    activePath?: string
    fixedTab?: boolean
    isFullPage?: boolean
  }
  children?: AppRouteRecord[]
}
```

注意：

- `component` 应为前端 `src/views` 下的组件路径，例如 `/system/user`。
- iframe 菜单必须设置 `meta.isIframe=true` 和 `meta.link`。
- 上线前应对 iframe link 做域名白名单控制。
- 菜单树只返回当前用户有权限访问的节点。

### 4.4 用户和角色接口建议

```text
GET    /api/user/list              用户分页
POST   /api/user                   新增用户
PUT    /api/user/{id}              更新用户
DELETE /api/user/{id}              删除用户
PATCH  /api/user/{id}/status       启用/停用用户

GET    /api/role/list              角色分页
POST   /api/role                   新增角色
PUT    /api/role/{id}              更新角色
DELETE /api/role/{id}              删除角色
GET    /api/role/{id}/permissions  查询角色权限
PUT    /api/role/{id}/permissions  保存角色权限
```

分页响应建议：

```ts
interface PageResult<T> {
  records: T[]
  current: number
  size: number
  total: number
}
```

### 4.5 文件上传接口建议

```text
POST /api/common/upload/image
POST /api/common/upload/file
POST /api/common/upload/wangeditor
```

```ts
interface UploadResponse {
  url: string
  fileName?: string
  size?: number
  contentType?: string
}
```

安全要求：

- 后端校验文件 MIME、后缀、大小。
- 图片重编码或存储到对象存储。
- 上传目录不允许直接执行脚本。
- 富文本上传返回的 URL 必须可信。

## 五、新增接口开发方式

### 5.1 新增类型

在 `src/types/api/api.d.ts` 中补充接口参数和响应类型：

```ts
declare namespace Api {
  namespace User {
    interface QueryParams {
      userName?: string
      current?: number
      size?: number
    }

    interface UserItem {
      id: number
      userName: string
      status: string
    }
  }
}
```

### 5.2 新增请求函数

在 `src/api/` 下按业务模块创建或更新文件：

```ts
import request from '@/utils/http'

export function fetchUserList(params: Api.User.QueryParams) {
  return request.get<PageResult<Api.User.UserItem>>({
    url: '/api/user/list',
    params
  })
}
```

### 5.3 页面中使用

表格类页面优先使用现有 `useTable`：

```ts
const { tableData, loading, pagination, search } = useTable({
  apiFn: fetchUserList,
  defaultParams: {
    current: 1,
    size: 10
  }
})
```

## 六、本地 Mock 方案

当前已接入 Vite middleware 本地 Mock：

```text
mock/local-mock.ts
```

开发环境通过 `.env.development` 控制：

```env
VITE_USE_LOCAL_MOCK = true
```

该值为 `true` 时，Vite 开发服务器会直接响应已覆盖的 `/api/*` 请求，不再转发到外部 Mock 服务。生产构建不会启用该 middleware。

当前已覆盖接口：

| 接口 | 方法 | 说明 |
| :--- | :--- | :--- |
| `/api/auth/login` | POST | 返回本地登录 token |
| `/api/user/info` | GET | 返回本地用户信息、角色、按钮权限 |
| `/api/user/list` | GET | 返回本地用户分页列表，支持基础查询和分页 |
| `/api/role/list` | GET | 返回本地角色分页列表，支持基础查询和分页 |
| `/api/v3/system/menus` | GET | 返回后端权限模式使用的本地菜单树 |
| `/api/article/types` | GET | 返回本地文章分类列表 |
| `/api/article/detail` | GET | 返回本地文章详情，用于替代外部 JSON |

Mock 维护规则：

- Mock 数据必须与后端接口契约一致。
- Mock 只用于开发和调试，不进入生产逻辑。
- 不在 Mock 中保存真实密码、真实 token、真实用户隐私。
- 菜单、角色、按钮权限 Mock 至少覆盖管理员和普通用户两类场景。
