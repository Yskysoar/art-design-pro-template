# 项目说明文档

最后更新：2026-06-04

## 一、项目定位

当前项目是基于 Art Design Pro 改造的前后端模板工程，用于持续补齐真实后端、权限体系、内容管理、系统管理和前台业务能力。当前开发重点是把原模板演示能力改造成可真实联调、可测试、可维护的业务模块。

项目根目录结构：

```text
Code/
├── art-design-pro/           # 当前前端项目
├── backend-template-java/    # 当前后端项目
├── art-design-pro-main/      # 原始前端参考快照，只读参考
└── art-design-pro-main.zip   # 原始参考压缩包
Docs/
Info/                         # 敏感信息目录，禁止读取、写入、提交
```

## 三、底层规划

### 3.1 架构分层

后端采用标准三层结构：

```text
Controller -> Service -> Mapper -> Database
```

约定：

- Controller 只处理参数、登录态、响应格式和入口权限。
- 真实业务规则放在 Service。
- 数据访问优先使用 MyBatis-Plus 内置能力。
- 新增或重构普通 JavaBean、Entity、DTO 时可优先使用 Lombok 减少 getter/setter、构造器等样板代码。
- Entity/DTO/VO 转换优先使用 MapStruct 编译期映射，复杂业务字段仍由 Service 显式计算后传入。
- 只有复杂查询、复杂统计或性能明确需要时才写自定义 SQL。

前端约定：

- API 请求函数放在 `src/api/`。
- 业务类型优先维护到 `src/types/api/api.d.ts` 或模块类型文件。
- 表格、表单、权限、请求封装优先复用已有组件和 hooks。
- 新增或调整页面、菜单、面包屑、工作标签页、弹窗、按钮、表单校验、空状态和提示信息时，必须同步维护中文与英文国际化文案。
- 路由与菜单标题优先使用 `menus.*` 国际化键；后端菜单和 Mock SQL 不应长期保存裸中文/裸英文标题。
- 当前项目不再依赖模板演示 Mock 作为生产逻辑。
- 生产部署时应复制 `Code/art-design-pro/.env.production.example` 为本机 `.env.production`，并将 `VITE_API_URL` 设置为真实后端地址；生产环境不得指向 Apifox 或本地 Mock。

命令与编码约定：

- Windows 下优先使用 PowerShell 7.6（`pwsh`）执行命令。
- 当前本机已验证 `pwsh` 版本为 7.6.2，输出编码为 `utf-8`，中文输出正常。
- 在 PowerShell 5.x 外壳中调用 `pwsh -NoLogo -NoProfile -Command '...'` 时，包含 `$变量` 的命令应使用单引号包裹，避免外层提前展开。

### 3.2 认证与权限

认证使用 Bearer JWT：

```text
Authorization: Bearer <token>
```

权限模式：

- 默认使用 `VITE_ACCESS_MODE=backend`。
- 后端返回当前用户可访问菜单树和按钮权限。
- 前端负责展示菜单、注册动态路由和按钮显隐。
- 后端接口必须独立做权限校验，不能只依赖前端显隐。
- 后端 CORS 通过 `app.security.cors.*` 配置，开发默认允许 `http://localhost:5173` 和 `http://127.0.0.1:5173`；生产必须使用 `CORS_ALLOWED_ORIGINS` 收敛为真实前端域名。
- 当前认证使用 Bearer JWT，不依赖 Cookie 会话，后端 CSRF 保持禁用；如果后续改为 HttpOnly Cookie 认证，必须同步补 CSRF 策略。

超级管理员：

- `R_SUPER` 可跳过普通权限码校验。
- 普通角色必须拥有对应权限码。

### 3.3 路由规划

动态路由依赖后端菜单树。后端菜单返回前端可识别的路由结构，前端注册时需要处理父子路径转换，避免登录后跳转 500 或刷新动态路由丢失。

隐藏路由用于：

- 个人中心。
- 文章详情。
- 社交聊天页 `/social/chat`。

隐藏路由必须出现在后端菜单或前端静态配置中，否则后端权限模式下刷新页面可能无法注册。

### 3.4 数据规划

当前已使用或规划的核心表：

| 模块 | 主要表 |
| :--- | :--- |
| 系统管理 | `sys_user`, `sys_role`, `sys_menu`, `sys_permission`, `sys_org`, `sys_config` |
| 认证账号 | 复用 `sys_user`、角色关系和组织关系 |
| 文章上传 | `article`, `article_category`, `article_attachment`, `file_resource` |
| 评论敏感词 | `article_comment`, `comment_sensitive_word` |
| 社交聊天 | `social_follow`, `social_block`, `social_conversation`, `social_message` |

### 3.5 Mock SQL 规划

Mock SQL 用于开发环境初始化真实数据库数据，不写入真实敏感信息。

规划：

```text
db/mock/
├── mock_001_system.sql
├── mock_002_article.sql
├── mock_003_sensitive_word.sql
└── mock_004_social.sql
```

每个业务模块完成后必须同步维护 schema、mock、接口文档和测试记录。
