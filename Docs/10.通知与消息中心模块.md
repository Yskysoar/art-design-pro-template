# 通知与消息中心模块

最后更新：2026-06-04

## 一、模块范围

通知与消息中心负责统一承接用户侧通知入口。

第一版已覆盖：

- 顶部铃铛未读红点。
- 顶部通知弹层读取真实通知。
- 消息中心隐藏路由 `/notification/center`。
- 分页查询通知。
- 单条标为已读。
- 全部标为已读。
- 关注、私信、评论回复事件写入通知。

第一版暂不做：

- WebSocket 实时推送。
- 系统公告后台发布。
- 通知模板管理。
- 站外通知，如邮件、短信、浏览器 Push。

## 二、数据模型

统一台账表：

```text
notification_message
```

核心字段：

```text
id, recipient_id, actor_id, notice_type, title, content,
target_type, target_id, target_url, read_time, create_time, deleted
```

通知类型：

| 类型 | 说明 |
| :--- | :--- |
| `SYSTEM` | 系统通知，预留 |
| `FOLLOW` | 有用户关注我 |
| `PRIVATE_MESSAGE` | 收到私信 |
| `COMMENT_REPLY` | 我的评论收到回复 |

## 三、事件来源

当前写入点：

- `SocialServiceImpl.follow`：关注成功后给被关注人写入 `FOLLOW`。
- `SocialServiceImpl.sendMessage`：私信发送成功后给接收人写入 `PRIVATE_MESSAGE`。
- `ArticleCommentServiceImpl.createComment`：回复评论成功后给父评论作者写入 `COMMENT_REPLY`。

同一事务内写入通知；通知写入异常会暴露并回滚主业务，便于开发阶段及时发现结构或数据问题。

## 四、接口

| 接口 | 方法 | 说明 |
| :--- | :--- | :--- |
| `/api/notifications` | GET | 当前用户通知分页 |
| `/api/notifications/unread-count` | GET | 当前用户未读通知数量 |
| `/api/notifications/{id}/read` | POST | 标记单条通知已读 |
| `/api/notifications/read-all` | POST | 标记当前用户全部通知已读 |

## 五、前端入口

- 顶部铃铛：显示未读红点，打开弹层后读取未读通知。
- 消息中心：`/notification/center`，主布局内隐藏路由。

页面能力：

- 按类型筛选。
- 按未读筛选。
- 查看通知列表。
- 单条已读。
- 全部已读。
- 跳转到目标页面。

## 六、Mock 数据

`mock_005_governance.sql` 已补充通知中心调试样例：

- `SYSTEM`：系统维护、文件资源台账更新、账号安全提醒。
- `FOLLOW`：关注与互关场景。
- `PRIVATE_MESSAGE`：文本、图片和附件私信触发的通知。
- `COMMENT_REPLY`：文章评论回复通知。
- 同时覆盖未读、已读、不同接收人、系统通知无触发人和跳转 URL。

## 七、后续计划

- 私信、关注和评论回复可接 WebSocket 实时推送。
- 系统公告发布能力可由运营后台扩展。
- 可增加通知去重策略，避免同一会话高频消息刷屏。
