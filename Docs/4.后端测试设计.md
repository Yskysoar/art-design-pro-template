# 后端测试设计

最后更新：2026-06-01

## 一、测试目标

后端测试需要保证当前模板后端在继续扩展业务模块时不会破坏已有认证、权限、菜单、配置项和通用响应契约。

优先目标：

- Service 层业务规则可独立验证，不依赖真实 MySQL。
- Controller 层接口契约可验证，包括认证、参数校验、响应结构和状态码。
- 数据库脚本可验证，避免 schema、mock SQL 和实体字段不一致。
- 前端已接入的真实接口在后端重启后可以快速回归。

## 二、测试分层

| 层级 | 工具 | 是否依赖数据库 | 主要验证 |
| :--- | :--- | :--- | :--- |
| 单元测试 | JUnit 5 + Mockito + AssertJ | 否 | Service 业务规则、边界条件、异常分支 |
| Web 层测试 | Spring Boot Test + MockMvc + spring-security-test | 否或弱依赖 | Controller 参数校验、认证拦截、统一响应 |
| 数据库集成测试 | Testcontainers MySQL 或本地测试库 | 是 | MyBatis-Plus 映射、SQL 脚本、分页、唯一约束 |
| HTTP 回归测试 | PowerShell/IDE HTTP Client | 是，依赖运行中后端 | 登录、菜单、用户、角色、配置项真实接口 |

| 删除后重建同名配置 | 逻辑删除记录不再占用活跃唯一键，允许重新新增同名 `configKey` |
| 新增内置配置项非法值 | 内置配置键只允许约定枚举或布尔值 |
| 更新可编辑配置项 | 可编辑配置允许修改业务字段和更新人 |
| 更新只读配置项 | `editable=0` 时拒绝修改 |
| 更新配置键 | 后端拒绝修改已有配置项的 `configKey` |
| 删除可编辑配置项 | 执行逻辑删除，不物理删除 |
| 删除系统内置配置项 | 系统内置配置不允许删除 |
| 删除不存在配置项 | 不存在时抛出业务异常 |

说明：配置项删除使用 MyBatis-Plus `deleteById`。`SysConfig.deleted` 使用 `@TableLogic(value = "0", delval = "id")`，删除后写入记录 ID，让后续查询自动排除非 0 数据，同时允许同名配置键重新占用 `deleted=0` 的唯一槽位。

## 八、真实接口回归建议

Agent 可自行启动后端后，按以下顺序做只读回归：

```powershell
$login = Invoke-RestMethod -Uri http://localhost:8080/api/auth/login -Method Post -ContentType 'application/json' -Body '{"userName":"admin","password":"admin123"}'
Invoke-RestMethod -Uri 'http://localhost:8080/api/config/list?current=1&size=20' -Method Get -Headers @{ Authorization = "Bearer $($login.data.token)" }
```

含义：先登录获取 JWT，再读取配置项分页列表。该命令不写数据库。

写接口回归建议先使用临时配置键，例如：

```text
dev_test_config_当前日期
```

测试后立即删除，避免污染长期 Mock 数据。
